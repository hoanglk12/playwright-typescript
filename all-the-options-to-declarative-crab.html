Technical Research Report — Expanding PLA GraphQL Suite to All GRA Sites

Date: 2026-06-07 | Branch: main | Status: Research complete — all decisions confirmed — ready for implementation

1. Summary

The existing suite (15 spec files, ~117 tests under tests/api/pla-*.spec.ts) targets one Accent Group Magento 2 GraphQL endpoint. The goal is to extend test coverage to SKX AU/NZ, DRM AU/NZ, and VAN AU/NZ — 6 additional brand+region combinations.

Headline finding — all brands share an identical GraphQL schema. Every brand exposes the same auth, cart, checkout, wishlist, loyalty, and createCustomer mutations with identical input shapes. Divergence is data + config only, not schema. This means the existing 15 specs can be reused for all brands with virtually no logic changes.

Two critical findings:

  1. "PLA NZ" is actually Platypus AU today. All GRA hosts default to store_code au, AUD, country AU when no Store header is sent. The existing PLA suite exercises the AU store on every host, including stag-platypus-nz. True NZ store views require a Store: nz header.

  2. A framework gap blocks NZ authenticated tests. ApiClient.init() applies auth via a mutually-exclusive switch — you cannot send a Bearer token AND a Store: nz custom header on one client instance. This is a hard blocker for NZ authed scenarios.

4. Schema Differences

  • Brand vs brand (PLA/SKX/DRM/VAN): IDENTICAL. All brand-specific and standard Magento 2 mutations present on every brand including loyalty_program_status, phone_number, applyRewardPointsToCart, applyQantasPointsToCart.
  • AU vs NZ: Store-view config only — currency (AUD→NZD), country codes (AU→NZ), available regions/shipping methods.
  • Qantas (user-confirmed): applyQantasPointsToCart is an AU-wide feature. All 4 AU brand sites include Qantas tests. All 4 NZ sites have Qantas tests removed entirely — not skipped, not gated, simply absent from the NZ spec files. No supportsQantas flag needed.

5. Options & Scoring

Option A — Duplicate per-site (90+ files)

Copy all 15 spec files 6 times, hardcode per-site data. Zero framework change, maximum isolation.

  • Cons: 90+ files, any fix to a shared scenario must be applied 6×, maintenance nightmare.
  • Score: 9/40

Option B — Parameterised loop (single spec, loop over sites array)

One spec file per scenario, sites.forEach(site => test.describe(site.name, ...)). No framework change.

  • Cons: Playwright serial mode + nested loops interact badly; poor isolation; CI can't run brands in parallel; report grouping messy.
  • Score: 23/40

Option C — Shared suite + Playwright project per site ⭐ RECOMMENDED

One spec file per scenario. Site-specific config injected via a SiteContext interface. Each brand gets its own data file under src/data/api/{brand}/. api.config.ts runs 8 projects (one per site), each setting GRAPHQL_BASE_URL and Store header. Full Playwright project isolation — independent reporting, parallel CI matrix.

Criterion Score
Maintenance (1-10) 9
Enterprise alignment (1-10) 9
SDET best practice (1-10) 9
CI impact (1-10) 9
Total 36/40

Option D — Brand-group overrides

Group similar brands, per-group spec overrides. Premise broken by uniform-schema evidence.

  • Score: 20/40

6. Recommended Approach — Option C (Phased)

Phase 1: AU brand hosts (no framework change)

createGraphQLClient({ baseURL: site.baseURL }) is already supported — the ...options spread in ApiTest.ts overrides baseURL. This covers all 4 AU brand hosts immediately.

New files:

  • src/data/api/sites.ts — SiteContext interface + registry of all 8 sites
  • src/data/api/skx/skx-au-data.ts — SKX AU test data
  • src/data/api/drm/drm-au-data.ts — DRM AU test data
  • src/data/api/van/van-au-data.ts — VAN AU test data
  • tests/api/skx-*.spec.ts (or refactor pla-* to be site-parameterised)
  • Re-key shared-state.ts to Map<siteCode, TestState>
  • Parameterise signInAndStoreToken to accept SiteContext

Changes to api.config.ts: add explicit projects pla-au, skx-au, drm-au, van-au (replacing the implicit default run).

Phase 2: NZ store views (requires framework fix)

Framework change needed in src/api/ApiClient.ts: make init() merge customHeaders additively with auth headers.

Why: ApiClient.init() uses a mutually-exclusive switch — AuthType.BEARER sets only the Bearer token; AuthType.CUSTOM sets only custom headers. You cannot apply both at once. NZ authenticated tests need two headers simultaneously on every request:

  • Authorization: Bearer {token} — for auth (cart, wishlist, checkout, loyalty, ~9/15 spec areas)
  • Store: nz — to activate the NZ store view

Without this fix, public NZ tests (catalog, search, storeConfig, countries) work via AuthType.CUSTOM, but all authenticated NZ scenarios are impossible. The fix is additive and backward-compatible — existing AU tests are unaffected.

New files:

  • src/data/api/pla/pla-nz-data.ts, src/data/api/skx/skx-nz-data.ts, etc.
  • Add NZ projects to api.config.ts with storeHeader: 'nz'

Phase 3: CI + reporting

  • Convert api-restful-tests.yml to a brand matrix (8 parallel jobs)
  • Per-site monocart trend files + Slack aggregation showing per-brand counts

11. Framework Analysis — Current State & Detailed Examples

Are Current Tests Running Correctly?

Yes, for PLA AU — the 15 existing specs are correct.

src/api/config/environment.ts:43 defaults graphqlBaseUrl to stag-platypus-au.accentgra.com. That host serves the AU store by default (no Store header required). Every PLA spec connects here, so AU data, AUD, and AU country codes are correct. Nothing is running wrongly today.

The Store View Header Gap — Exactly Where It Breaks

The problem is future NZ tests, not existing AU ones. Here is the specific failure path.

Step 1 — NZ hostname does NOT mean NZ store:

# Without Store header → AU store (default on all NZ hostnames)
POST https://stag-platypus-nz.accentgra.com/graphql
→ { "store_code": "au", "base_currency_code": "AUD" }

# With Store: nz header → NZ store
POST https://stag-platypus-nz.accentgra.com/graphql
Store: nz
→ { "store_code": "nz", "base_currency_code": "NZD" }

Step 2 — The framework cannot send Bearer + Store header together.

The init() method in src/api/ApiClient.ts:51-73 uses a mutually exclusive switch — exiting each case with break before customHeaders can be applied:

// ApiClient.ts — current (broken for NZ authenticated tests)
switch (this.clientOptions.authType) {
  case AuthType.BEARER:
    headers['Authorization'] = `Bearer ${this.clientOptions.token}`;
    break;   // exits here; customHeaders are NEVER applied
  case AuthType.CUSTOM:
    Object.assign(headers, this.clientOptions.customHeaders);
    break;   // sets Store: nz but NO Authorization header
}

NZ authenticated tests need both simultaneously:

  • Authorization: Bearer eyJhb... — for cart / wishlist / checkout / loyalty (~9/15 spec areas)
  • Store: nz — to activate NZ store view on every request

With the current code, only one can be set. AuthType.BEARER silently ignores customHeaders. AuthType.CUSTOM sets Store: nz but adds no auth token.

Concrete effect in a test:

// What you would write for an NZ authenticated test today:
const nzAuthClient = await createGraphQLClient({
  authType: AuthType.BEARER,
  token: customerToken,
  customHeaders: { 'Store': 'nz' },  // silently dropped — never sent
});
// Request that actually goes out:
//   Authorization: Bearer eyJhb...  ✓
//   Store: nz                       ✗  (missing)
// Magento responds with AUD/AU data, not NZD/NZ

For public NZ tests (catalog, search, storeConfig — no auth needed), AuthType.CUSTOM with { 'Store': 'nz' } does work. Those 6/15 spec areas are unblocked even before the fix.

Phase 2 — The ApiClient.init() Fix (Required for All NZ Authenticated Tests)

One additive merge after the switch. Backward-compatible — existing AU tests are unaffected because they do not set customHeaders.

// ApiClient.ts — fixed init()
async init(): Promise<void> {
  const headers: Record<string, string> = {};

  switch (this.clientOptions.authType) {
    case AuthType.BEARER:
      if (this.clientOptions.token) {
        headers['Authorization'] = `Bearer ${this.clientOptions.token}`;
      }
      break;
    // ... other cases unchanged ...
  }

  // NEW: always merge customHeaders additively, regardless of authType
  if (this.clientOptions.customHeaders) {
    Object.assign(headers, this.clientOptions.customHeaders);
  }

  this.context = await request.newContext({ extraHTTPHeaders: headers, ... });
}

After this fix, an NZ authenticated client works correctly:

// After fix — Bearer + Store: nz both sent on every request
const nzAuthClient = await createGraphQLClient({
  baseURL: 'https://stag-platypus-nz.accentgra.com/graphql',
  authType: AuthType.BEARER,
  token: nzCustomerToken,
  customHeaders: { 'Store': 'nz' },
});
// Authorization: Bearer ...  ✓
// Store: nz                  ✓
// Magento returns NZD, NZ country codes, NZ shipping methods
