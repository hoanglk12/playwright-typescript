Composition-Based Page Object Model

Favour composition over inheritance for all browser interactions
AbstractBasePage → Composed Helpers Architecture

All page objects extend BasePage, which owns 8 specialised helper instances as protected readonly fields. Browser interactions are delegated to helpers — never call page.locator() or page.click() directly inside a page class. This decouples interaction logic from page semantics and makes helpers independently testable and swappable. 

// src/pages/base-page.ts — abstract root of the POM hierarchy
export abstract class BasePage {
  protected page: Page;
  protected readonly waits:    WaitHelper;
  protected readonly elements: ElementHelper;
  protected readonly style:    StyleHelper;
  protected readonly frames:   FrameHelper;
  protected readonly files:    FileHelper;
  protected readonly storage:  StorageHelper;
  protected readonly network:  NetworkHelper;
  protected readonly tables:   TableHelper;

  constructor(page: Page) {
    this.page     = page;
    this.waits    = new WaitHelper(page);
    this.elements = new ElementHelper(page, this.waits);
    this.style    = new StyleHelper(page, this.waits);
    this.frames   = new FrameHelper(page);
    this.files    = new FileHelper(page, this.waits);
    this.storage  = new StorageHelper(page);
    this.network  = new NetworkHelper(page);
    this.tables   = new TableHelper(page, this.waits, this.elements);
  }
}
Locators Hoisted to Class Fields Convention

All Locator instances and raw selector strings must be declared as private readonly class fields at the top of every page class. This makes selectors discoverable in one place, enables refactoring without touching method bodies, and satisfies TypeScript's strict mode. Dynamic locators (parameter-driven) live in private helper methods that reference field-level constants.

✅ Correct
class CheckoutPage extends BasePage {
  // Locators as class fields
  private readonly submitBtn =
    this.page.getByRole('button',
      { name: 'Place Order' });
  private readonly itemSel =
    '[data-testid="cart-item"]';

  async placeOrder(): Promise<void> {
    await this.elements.clickLocator(
      this.submitBtn);
  }
}
❌ Banned
class CheckoutPage extends BasePage {
  async placeOrder(): Promise<void> {
    // WRONG — inline selector in helper call
    await this.elements.clickElement(
      '[data-testid="submit"]');

    // WRONG — direct page.locator
    await this.page
      .locator('.btn-submit')
      .click();
  }
}
🚫
Banned selectors: Hierarchical structural selectors like div > span > ul > li:nth-child(2) are prohibited. They break on any DOM restructure and carry no semantic meaning. Prefer getByRole(), getByLabel(), getByText(). CSS selectors are only acceptable for computed-style queries (this.style.*) or evaluate() calls.

Custom Fixtures — Playwright Extension Pattern

Page objects and shared utilities injected via test.extend()
test.extend<CustomFixtures> Design Pattern

src/config/base-test.ts extends Playwright's test with all page objects and utilities. Tests receive page objects via destructuring — no new calls in spec files. Each fixture is lazily instantiated per test, ensuring isolation. The use() function defines setup and teardown boundaries (code after await use(...) is teardown). 

// src/config/base-test.ts
export const test = base.extend<CustomFixtures>({
  homePage: async ({ page }, use) => {
    await use(new HomePage(page));
  },

  softAssert: async ({}, use) => {
    await use(new SoftAssertHelper(new TestLogger(base.info().title)));
  },

  ecommerceCartOverlayPage: async ({ page }, use) => {
    await use(new EcommerceCartOverlayPage(page));
    // Teardown: Firefox SPA service-worker workaround (see §23)
    if (page.context().browser()?.browserType().name() === 'firefox') {
      await page.goto('about:blank', { waitUntil: 'commit' }).catch(() => {});
    }
  },
});

export { expect };
export const softExpect: typeof expect.soft = expect.soft.bind(expect);

Test Data Management

Typed, generated, never hardcoded in specs
Factory + Typed Interface Pattern Pattern

All test data lives in src/data/. Every exported constant and generator return type must carry a named interface annotation — never rely on TypeScript inference for exported data shapes. Static constants use const objects; dynamic/random data uses generator classes with explicit return types. 

// ✅ Correct — named interfaces, annotated const, typed generator
export interface LoginCredentials { username: string; password: string; }

export interface AdminDataShape {
  VALID_ADMIN:    LoginCredentials;
  INVALID_ADMIN:  LoginCredentials;
  EMPTY_CREDENTIALS: LoginCredentials;
}

export const AdminData: AdminDataShape = {
  VALID_ADMIN:   { username: 'admin', password: 'pass123' },
  INVALID_ADMIN: { username: 'wrong', password: 'wrong'  },
  EMPTY_CREDENTIALS: { username: '', password: '' },
};

export class AdminTestDataGenerator {
  static generateCustomerData(): CustomerData { // ← explicit return type
    const ts = Date.now();
    return { customerName: `Test ${ts}`, email: `test${ts}@mail.com` };
  }
}
LocationContent
src/data/*.tsUI / general test data modules
src/data/api/API test data — one file per feature domain (pla-catalog-data.ts, pla-auth-data.ts…)
src/data/ecommerce/storefronts.tsEcommerce brand/storefront config constants
src/data/api/sites.tsGRA brand registry — SiteContext per brand with baseURL, testData, hasLoyalty flag
Interface vs Factory — Two Separate Concepts Clarification

The pattern name "Factory + Typed Interface" is two distinct things working together — not one concept. The interface is only a shape contract: it declares what fields exist and their types at compile time. It creates nothing. The factory is the function or class method that generates dynamic values at runtime and returns a fully populated object that satisfies the interface. Defining an interface with typed properties is not the factory — it is only the "Typed Interface" half.

Interface — shape contract only (compile-time)
// Declares what fields must exist and their types
// Creates NOTHING — no values, no logic
export interface GraTestData {
  validCustomer:    CustomerInput;
  validCredentials: ValidCredentials;
  invalidCartId:    string;
}
Factory — generates values and builds the object (runtime)
// Generates dynamic values FIRST
export function createBrandTestData(prefix: string): GraTestData {
  const email = `${prefix}test${Date.now()}@mail.com`;
  const name  = generateFirstName();
  // Returns a REAL OBJECT — all fields use the same values
  return { validCustomer: { email, firstname: name, ... },
           validCredentials: { email, ... } };
}
ℹ️
Why the factory generates values before the return: all fields for the same logical entity (email, name, address) must share the same randomly-generated values. If email were generated separately inside validCustomer and again inside validCredentials, two different Date.now() calls would produce two different emails — the credentials would no longer match the registered customer.
When to Use Factory vs Static Const Decision Guide

Not every data file needs a factory. The choice depends entirely on whether the data must change per test run.

Data typeUseReason
Customer email, name, addressFactoryMust be unique per run — registering the same email twice fails on a live backend
Brand URLs, nav labels, country codesStatic constFixed — Australia's country code is always 'AU', no generation needed
Error messages, expected stringsStatic constFixed expected values used for assertion — must not change between runs
Invalid/negative test inputsStatic constDeliberate bad values — randomising them would make negative tests unpredictable

Assertion Patterns — Hard vs Soft

Strategic choice between fail-fast and accumulate-all-failures
Pattern A — softExpect
Drop-in replacement for expect. Does NOT log internally — safe to pair with logger.verify() before it.

import { softExpect } from '@config/base-test';
softExpect(title).toContain('Expected');
Pattern B — softAssert fixture
Integrates with TestLogger. Each call logs 🔵 [SOFT] internally. Do NOT also call logger.verify() before it — duplicate log entry.

softAssert.toBe(count, 12, 'Item count');
await softAssert.toBeVisible(el, 'Header');
When to Use Each Type Decision Guide
ScenarioUseReason
Multiple independent checks on same page stateSoftSee all failures at once
Per-item assertions inside a loopSoftAll items reported, not just first failure
Precondition that guards subsequent stepsHardFollowing steps are meaningless if this fails
Playwright locator assertions (expect(locator).toBeVisible())HardSoftAssertHelper has no locator equivalent
expect.poll()HardHas its own retry/timeout logic; soft adds no value
Single-assertion testHardSoft provides no benefit with one check

Structured Test Logging

Observability for every test step, attached to HTML reports
TestLogger — Step / Action / Verify Triad Pattern

Every test creates a TestLogger via createTestLogger(testName). Logs are written to ./test-logs/test-execution.log and also attached as annotations to the Playwright HTML report, making failure context available directly in the report without opening log files. 

const logger = createTestLogger('TC_01 Login with valid credentials');

logger.step('Step 1 - Navigate to login page');   // 🔍 numbered step
logger.action('Click', 'Submit button');        // ⚡ interaction
logger.verify('URL contains /dashboard',         // ✅ assertion outcome
  '/dashboard', page.url());
logger.error(err, 'Sign-in mutation failed');   // ❌ error context

AuthType Strategy Enum API Pattern

Auth strategies are expressed as an enum, never as string literals. Each value triggers different header construction inside ApiClient.init(). New strategies are added by extending the enum — no switch-statement sprawl in test code.

ValueHeader produced
AuthType.NONENo auth headers
AuthType.BASICAuthorization: Basic <base64>
AuthType.BEARERAuthorization: Bearer <token>
AuthType.API_KEYCustom header name + value
AuthType.CUSTOMArbitrary customHeaders map
Static Token Store — Cross-Test Auth Sharing API Pattern

ApiClient.tokenStore is a static Record<string, string>. Tokens persist across all tests within a single worker process. Use ApiClient.storeToken(key, token) after login; retrieve with ApiClient.getToken(key) in later tests.

Factory Fixtures — Dynamic Client Creation at Test Time Factory Pattern

ApiTest.ts exposes four factory fixtures alongside the standard singleton fixtures (apiClient, graphqlClient, etc.). A factory fixture is not a pre-built client — it is a function the test calls to create one or more clients on demand, each with custom options merged on top of environment defaults. The fixture itself manages the full lifecycle: it initialises every client the test creates and disposes them all after use() completes — the test body never calls init() or dispose().

Factory fixtureReturnsWhen to use over the singleton
createClientApiClientNeed a low-level client with non-default auth or headers
createClientExtApiClientExtNeed a wrapper-chain client with Bearer token or API key
createRestfulApiClientRestfulApiClientNeed a device API client with custom timeout or auth
createGraphQLClientGraphQLClientNeed a GraphQL client pointed at a different brand baseURL (GRA pattern)
Factory fixture internals (createClientExt)
// src/api/ApiTest.ts:118–130
createClientExt: async ({ apiBaseUrl }, use) => {
  const clients: ApiClientExt[] = [];   // ① lifecycle tracker

  const factory = async (
    options: Partial<ApiClientOptions>
  ): Promise<ApiClientExt> => {
    const client = new ApiClientExt({
      baseURL: apiBaseUrl, ...options  // ② merge defaults + overrides
    });
    await client.init();               // ③ always initialised
    clients.push(client);              // ④ registered for teardown
    return client;
  };

  await use(factory);                 // ⑤ hand factory to the test

  for (const c of clients)
    await c.dispose();               // ⑥ dispose ALL created clients
},
Usage in a test
// Singleton — one client, default auth, no options
test('TC_01', async ({ apiClientExt }) => {
  const r = await apiClientExt.getWithWrapper('/public');
  await r.assertStatus(200);
});

// Factory — caller specifies only the delta
test('TC_05', async ({ createClientExt }) => {
  const authed = await createClientExt({
    authType: AuthType.BEARER,
    token: storedToken,         // only override what differs
  });
  const r = await authed.getWithWrapper('/protected');
  await r.assertStatus(200);
  // no dispose() — factory fixture handles it
});
⚠️
Factory fixture vs singleton fixture: The standard fixtures (apiClient, apiClientExt, graphqlClient) are singletons — one instance per test, default auth, no custom options. Use the factory fixtures when a test needs non-default auth, multiple independent clients, or a client pointed at a different base URL (e.g. GRA brand URL via createGraphQLClient({ baseURL: site.baseURL })). Without factory fixtures, the test would need manual new Client(...) + init() + try/finally dispose() — the factory removes all three.

GraphQL Testing Patterns

Safe variable passing, error guards, staging tolerance
✅ Safe — variables as second arg
await graphqlClient.queryWrapped(
  `query GetUser($id: ID!) {
    user(id: $id) { name }
  }`,
  { id }   // ← variables object
);
❌ Banned — string interpolation
// Injection risk + breaks query caching
await graphqlClient.queryWrapped(
  `query { user(id: "${id}") { name } }`
);
Error Presence Guard
Always use if (gql.errors?.length), never if (gql.errors). An empty array [] is truthy — the bare check silently misclassifies a failed response as success.
Staging Price-Range Tolerance
Catalog queries use assertNoCriticalErrors(gql) instead of assertNoErrors(). This tolerates price_range path errors (broken staging data) while failing on all other error types.
Schema Gap Handling
Operations absent from staging schema (productSearch, guestOrder) are detected by checking for "Cannot query field" in errors and returning early — test is not marked failed.
GraphQL Batching
GraphQLClient supports query batching via addToBatch() + executeBatch(). Configurable maxBatchSize. Used for introspection and bulk operations.

GRA Multi-Brand Test Pattern

One shared spec suite running across 4 brand endpoints via Playwright projects
Project-Per-Brand with Metadata Injection Design Pattern

api.config.ts declares one Playwright project per brand (pla-au, skx-au, drm-au, van-au). Each project sets metadata: { siteCode: '...' }. All 15 pla-*.spec.ts files are shared across all brand projects unchanged. The graTest fixture reads testInfo.project.metadata.siteCode to inject the correct SiteContext (baseURL, testData, currency, feature flags). 

// api.config.ts — one project per brand
projects: [
  { name: 'pla-au', testMatch: ['**/pla-*.spec.ts'], metadata: { siteCode: 'pla-au' } },
  { name: 'skx-au', testMatch: ['**/pla-*.spec.ts'], metadata: { siteCode: 'skx-au' } },
  { name: 'drm-au', testMatch: ['**/pla-*.spec.ts'],
    testIgnore: ['**/pla-loyalty-rewards.spec.ts'], // feature flag
    metadata: { siteCode: 'drm-au' } },
],

// gra-test.ts — site context injected into every fixture call
const site = siteRegistry[testInfo.project.metadata.siteCode]; // SiteContext
const graphqlClient = await createGraphQLClient({ baseURL: site.baseURL });
Always-Fresh Auth
Every beforeAll calls signInAndStoreToken(client, logger, site, siteState). Never reuse a cached token — prior test runs (e.g. TC_01 revoking tokens) invalidate stale strings.
Always-Fresh Cart
Cart creation happens in beforeAll, never in a test. Playwright retries re-run beforeAll but NOT earlier tests, so a cartId set in a prior test is empty string on retry.
Guest Carts for Negative Tests
Magento's createEmptyCart for authenticated users returns the existing active cart. Negative tests (missing shipping / missing payment) use unauthenticated guest clients to guarantee a genuinely incomplete cart.
Safe SKU Discovery
Candidate SKU lists collect only confirmed IN_STOCK SimpleProduct or ConfigurableProduct variant SKUs. No else if (item.sku) fallback — that grabs unaddable parent configurable SKUs.

Shared State — Map-Keyed Singleton

Per-brand isolated state without cross-contamination between workers
TestState + getStateForSite(siteCode) Pattern

tests/api/shared-state.ts provides a TestState singleton class keyed by siteCode. When 4 brands run concurrently across 4 workers, each worker has its own Node.js process — the Map is never shared across processes. Within a worker, the Map guarantees each brand's token, cartId, addressId are isolated. Setters throw on empty/falsy values to prevent silent blank-state bugs. 

// Multiple brands — each gets its own bucket
const pla = getStateForSite('pla-au');
const skx = getStateForSite('skx-au');

pla.setCustomerToken(token);   // only affects pla-au
skx.setCartId(cartId);         // only affects skx-au

Parallelism & Worker Configuration

Different strategies for UI tests vs API tests
ConfigfullyParallelworkersRationale
playwright.config.ts
(UI tests)		 true WORKERS env || "50%" Tests are stateless (each gets a fresh browser context). 50% of CPU cores by default balances speed and resource use. Override per-run with WORKERS=4 or WORKERS=100%.
api.config.ts
(API tests)		 false 4 (fixed) 4 brands run concurrently (one worker per brand). Within each worker, tests run sequentially — required because module-level let state (cartId, token) is shared across tests in the same file. Do NOT set fullyParallel: true.
⚠️
Serial mode removed from GRA specs (2026-06-11): test.describe.configure({ mode: 'serial' }) was deliberately removed from all 15 pla-*.spec.ts files. fullyParallel: false alone guarantees sequential ordering without cascade-skip-on-failure. Serial mode would cascade-skip all remaining tests when any one fails — hiding test signal.
Dynamic Worker Expression
"50%" uses half the CPU cores (minimum 1). A numeric string like "4" fixes the count. The WORKERS env var overrides the default — used by CI matrix to tune per-OS.
CI Matrix: Windows + macOS
UI test job runs on both windows-latest and macos-latest simultaneously (fail-fast: false). This catches platform-specific issues (e.g. Firefox path-handling differences).
Firefox Skipped in CI Ecommerce
Firefox project has testIgnore: ['**/ecommerce/smoke/**'] in CI. Ecommerce smoke tests run Chromium-only in CI to reduce flakiness from SPA service-worker teardown issues on Firefox.

Retry Mechanism

Layered retry strategy: test-level, hook-level, and poll-based
Test-Level Retries
retries: process.env.CI ? env.retries : 0
UI: 2 retries in CI, 0 locally.
API: 2 retries in CI, 0 locally.
Retries re-run the full test including its beforeAll.
Trace on First Retry
trace: "on-first-retry" (default).
Playwright records a full trace (DOM snapshots, network, console) on the first retry attempt. Zero overhead for passing tests; full debugging context for flaky ones.
Screenshots on Failure
screenshot: "only-on-failure".
Videos: "retain-on-failure".
Both configurable per-environment via SCREENSHOT_MODE and VIDEO_MODE env vars.
Poll-Based Retry (Eventual Consistency)
expect.poll(() => condition, { timeout, intervals })
Used for staging eventual-consistency gaps (e.g. token revocation lag). Also exposed as TIMEOUTS.POLL_INTERVAL_NORMAL (1s) wait + re-query loops.
SKU Discovery Retry Loop
beforeAll tries multiple search terms and multiple SKU candidates in a loop. Each candidate's addProductsToCart call checks both errors?.length and user_errors — only moves on when a SKU successfully lands in the cart.
PDP Size Selection Retry
Ecommerce PDP tests retry navigating to products when the discovered size is out of stock. Max attempt limit prevents infinite loops; OOS products are treated as expected cases, not failures.

workflow_dispatch Inputs
Environment, mode, workers, browser, test type are all selectable at trigger time. Enables ad-hoc single-browser or single-environment runs without code changes.
Shard Merging
OS-specific JSON reports are downloaded as artifacts and merged in a test-report job. Merged counts feed Slack and GitHub Step Summary. Monocart's JSON is used for trend history via branch-scoped actions/cache.

Rich Slack message with colour-coded status, per-suite counts (total/passed/failed/skipped/flaky/duration), environment and browser context, and action buttons (Workflow Run, UI Report, API Report, View Commit).

Previous run's index.json is cached per-branch with actions/cache. When MONOCART_TREND_FILE env var points to it, monocart renders a pass/fail trend chart across runs in the interactive report.

Visual Regression Testing

Percy integration with silent skip when token is absent
Conditional Percy Snapshots Pattern

Percy snapshots only execute when PERCY_TOKEN is set in the environment. When absent (local development), percyHelper.snapshot() silently skips — the test passes without any visual assertion. This means visual tests can live in the same suite as functional tests without breaking local runs. PercyHelper is NOT a BasePage field — it is only available via the percyHelper fixture.

TypeScript Quality Standards

Strict mode, explicit typing, linting enforced in CI
Strict Mode
"strict": true in tsconfig.json. Enables strictNullChecks, noImplicitAny, strictFunctionTypes. CI runs npm run lint (tsc --noEmit) — type errors fail the build.
Explicit Return Types
All exported functions and generator methods declare explicit return types. Generator classes return typed interfaces, never inferred. This prevents interface drift when implementations change.
No any Casts
AuthType.BEARER enum is always used — never "bearer" as any. Data shapes carry named interfaces. page.evaluate() receives typed arrow functions, not raw strings where avoidable.
Optional Chaining on Error Arrays
gql.errors?.[0]?.message ?? '' even inside a length check. TypeScript does NOT narrow errors to non-undefined inside if (gql.errors?.length) — optional chaining is still required on index access.
Module-Level let Initializers
let cartId: string = '' not let cartId: string. Explicit initializers prevent TypeScript's definite assignment checks from being ambiguous and make the "empty state" semantic explicit.
Module Paths: NodeNext
"module": "NodeNext" + "moduleResolution": "NodeNext" aligns TypeScript's module resolution with Node.js ESM semantics, avoiding common .js extension and re-export issues.

API Mocking for UI Tests

Centralised mock scenarios, network interception via Playwright route API
ApiMockService — Centralised Scenarios Pattern

src/api/ApiMockService.ts wraps Playwright's page.route() API into named scenario methods. Tests call high-level methods like mockSuccessfulLogin() instead of managing route handlers directly. NetworkHelper provides the low-level route/unroute/intercept primitives; ApiMockService composes them into reusable scenarios. 

const mocks = new ApiMockService(page);

await mocks.mockSuccessfulLogin();          // stub auth endpoint
await mocks.mockProductList(productData);    // stub catalogue
await mocks.mockGraphQLQuery('GetCart', {}); // stub specific GQL op
await mocks.mockGraphQLError('PlaceOrder');  // simulate error

Browser-Specific Workarounds

Documented fixes for known browser protocol / SPA issues
Firefox SPA Service-Worker Teardown Workaround

All six ecommerce fixtures (ecommerceHomePage, ecommerceNavPage, ecommerceSearchPage, ecommercePLPPage, ecommercePDPPage, ecommerceCartOverlayPage) navigate to about:blank before teardown on Firefox. Root cause: Firefox's Juggler protocol hangs on context.close() when SPAs have active service workers or persistent WebSocket/analytics connections. Navigating to about:blank triggers the unload lifecycle — deregistering service workers and closing connections — before the context is destroyed. Do NOT remove this workaround from any of these fixtures. 

// Fixture teardown (after await use(...))
if (page.context().browser()?.browserType().name() === 'firefox') {
  await page.goto('about:blank', { waitUntil: 'commit', timeout: 5000 })
    .catch(() => {});  // swallow — page may already be closing
}
Firefox User Preferences for Stability Workaround

The Firefox project in playwright.config.ts sets explicit firefoxUserPrefs to disable geolocation, push notifications, background processes, and speculative connections that can cause flaky network waits or test-isolation issues. Network timeouts are also tightened to prevent lingering connections from blocking context teardown.

Staging Eventual-Consistency Polling Workaround

PLA staging token revocation has a race condition: after revokeCustomerToken, the old token may still be accepted for 1–5s. Tests poll the protected resource with TIMEOUTS.POLL_INTERVAL_NORMAL (1s) intervals until the graphql-authorization error appears, rather than asserting immediately after revocation.
