Refactoring Recommendations for Shared State Module

The current shared-state.ts implementation has several issues that could lead to test pollution and debugging challenges:

Current Issues:

  • Global Variables: The use of global variables (customerToken, customerId, etc.) creates a shared mutable state across tests, which can cause race conditions and make tests non-deterministic.
  • Lack of Encapsulation: There are no access controls or validation mechanisms - any test file can modify these values directly.
  • No Initialization Checks: There's no mechanism to ensure values are properly initialized before use.

Recommended Refactoring:

// shared-state.ts (refactored)
class TestState {
  private _customerToken: string | null = null;
  private _customerId: string | null = null;
  private _cartId: string | null = null;
  private _addressId: string | null = null;

  // Private setter with validation
  private setCustomerToken(token: string): void {
    if (!token || typeof token !== 'string') {
      throw new Error('Invalid customer token');
    }
    this._customerToken = token;
  }

  // Public getter with validation
  getCustomerToken(): string {
    if (!this._customerToken) {
      throw new Error('Customer token not set');
    }
    return this._customerToken;
  }

  // Reset method for test cleanup
  reset(): void {
    this._customerToken = null;
    this._customerId = null;
    this._cartId = null;
    this._addressId = null;
  }

  // Factory method for singleton instance
  static getInstance(): TestState {
    if (!TestState.instance) {
      TestState.instance = new TestState();
    }
    return TestState.instance;
  }

  private static instance: TestState | null = null;
}

export default TestState.getInstance();

This refactored version:

  • Uses encapsulation to prevent direct access to internal state
  • Adds validation checks for input data
  • Provides a reset method for proper test cleanup
  • Uses a singleton pattern to ensure consistent state management
  • Throws meaningful errors when invalid operations occur

Suggestions for Improving Test Naming Conventions

Current naming follows a pattern like E2E-CART-001-XXX but could be more descriptive:

E2E-CART-001-001 - Skechers AU Mini cart shows 0 items on fresh session

Improved:

Cart_Smoke_InitialEmpty_ShouldShowZeroItemsOnFreshSession

Best Practices:

  • Use PascalCase for test names
  • Include the feature area, test type, and expected behavior
  • Follow the format: [Feature]_[TestType]_[ExpectedBehavior]
  • Avoid special characters and spaces in test names
  • Make test names self-documenting so they don't require reading the implementation
