DummyJSON API Test Scaffold

Adding a fourth API practice target — dummyjson.com (products / carts / users / JWT auth) — to the existing REST + GraphQL suite, following the same service-class and fixture conventions already used for restful-booker and restful-api.dev.

Status: Planned · Reviewed 2026-07-14 New files: 8 Edited files: 4–5 Est. tests: ~26

·Context

The user asked for a practice API test suite scaffolded against dummyjson.com, chosen because its data model (products, carts, users) is closer to real e-commerce shape than the framework's existing practice targets, and it adds JWT-based auth (accessToken/refreshToken) as a new pattern to exercise. This mirrors two existing integrations — RestfulBookerService (login + token reuse) and RestfulApiClient (plain CRUD) — but combines both patterns across four related resource domains sharing one base URL and one auth flow.

The endpoint contracts below were verified live against dummyjson.com/docs before writing this plan, so the scaffold isn't guessing at field names.

Scope decisions (confirmed with the user): four domain-split spec files, mirroring the repo's existing gra-catalog.spec.ts / gra-cart-minicart.spec.ts / gra-customer-profile.spec.ts / gra-authentication.spec.ts split — and fuller CRUD coverage (~5–7 tests per domain) rather than a bare smoke scaffold.

·Review Amendments (2026-07-14)

A technical-research review verified every structural claim in this plan against the live codebase (mirror targets, fixture patterns, faker, ApiResponseWrapper, misc-api testMatch) and the dummyjson.com contract. The plan was endorsed with the following amendments, now folded into the sections below:

  • Bearer-header gap (blocking): ApiClient.get() (src/api/ApiClient.ts:143) accepts only (url, queryParams?) — no headers parameter, unlike post/put/patch. The original getAuthenticatedUser() could not attach Authorization: Bearer to GET /auth/me. Fixed via ApiClient.withStoredToken() (Option B — no shared-file change); see the service card.
  • Default execution mode, not serial: test.describe.configure({ mode: 'serial' }) removed from all four specs. Tests are independent (mutations are simulated), fullyParallel: false already serializes within a file, and serial mode causes cascade-skips that hide signal.
  • Describe titles contain the literal DummyJSON so the --grep "DummyJSON" verification command matches.
  • Casing: dummyjsonBaseUrl is used for both the env field and the fixture. The existing pattern has split casing (restfulApiBaseUrl env vs restfulApiBaseURL fixture); this plan deliberately uses one consistent form rather than mirroring the split.
  • CI cosmetic edit added: the misc-suite step-summary string in api-restful-tests-with-slack.yml hardcodes the spec list and must mention the dummyjson specs; see Wiring.

CartsGET /carts, /carts/{id}, /carts/user/{userId}, POST /carts/add, PUT /carts/{id}, DELETE /carts/{id}
UsersGET /users, /users/{id}, /users/search?q=, POST /users/add, PUT /users/{id}, DELETE /users/{id}
List response shape{ <resource>: [...], total, skip, limit }
Design-critical quirk: add/update/delete endpoints on dummyjson are simulated — they return a plausible response (new id, isDeleted / deletedOn, etc.) but do not persist server-side. Every mutating test must assert against the response of the mutating call itself and must never re-fetch afterward to verify persistence (unlike objects-crud.spec.ts's real-backend cleanup pattern against restful-api.dev).

Amended — Bearer header on GET: ApiClient.get() has no headers parameter (only post/put/patch do), so /auth/me cannot attach Authorization: Bearer through the inherited get(). getAuthenticatedUser() instead spins up a pre-authed client via the existing ApiClient.withStoredToken() helper — contained to this service, no change to the shared base class: 
async getAuthenticatedUser(): Promise<ApiResponseWrapper> {
  const client = await ApiClient.withStoredToken(
    { baseURL: this.baseUrl, timeout: this.timeoutMs },
    this.tokenKey
  );
  try {
    return new ApiResponseWrapper(await client.get('/auth/me'));
  } finally {
    await client.dispose();
  }
}

Every method wraps this.get/post/put/delete(...) in new ApiResponseWrapper(response) (exported from src/api/ApiResponse.ts) — specs never touch a raw apiClient/apiClientExt directly. Note ApiClient.delete(url, data?) also takes no headers — fine here, since dummyjson deletes are unauthenticated.
