OpenTelemetry Demo (Astronomy Shop) — API Test Suite

A 3-group Playwright TypeScript API test suite against the running OTel Demo stack, source-verified and live-confirmed before any test code was written.

Context

The goal was a Playwright TypeScript API test suite against the OpenTelemetry Demo microservices app (gateway at http://localhost:8080, started via docker compose -f compose.yaml -f compose.full.yaml -f compose.observability.yaml up), organized into 3 groups — functional, integration, fault-injection — partly intended as interview talking points about microservice testing strategy.

The explicit brief was: verify everything against source first, then confirm live with curl, and flag any discrepancy rather than silently writing a test around a guessed behavior. That discovery pass is what this document captures. Several real discrepancies turned up between the original task assumptions and actual system behavior — those corrections are what shaped the final Phase 2 design.

Repo location: the actual OpenTelemetry Demo repo lives at E:\OLDDATA\DATA\TESTING\MicroserviceTesting\opentelemetry-demo (confirmed via docker inspect frontend-proxy → compose working_dir), not inside this playwright-typescript framework repo. The new test project will be created at E:\OLDDATA\DATA\TESTING\MicroserviceTesting\otel-demo-api-tests, sibling to the otel-demo repo — this HTML plan is filed here in docs/ purely as this framework's documentation convention.

C. flagd-ui flag toggle mechanism — corrected understanding

src/flagd-ui is an Elixir Phoenix LiveView app, not a REST-first frontend. Its dashboard flips flags over a WebSocket (phx-change event), but a plain JSON HTTP API also exists and is what the test suite uses:

  • GET http://localhost:8080/feature/api/read → {"flags": {...entire config...}} live-confirmed
  • POST http://localhost:8080/feature/api/write with body {"data": <entire modified flags JSON>} — replaces the whole file, no single-flag endpoint exists over HTTP. flagd-ui writes straight to src/flagd/demo.flagd.json (bind-mounted into both flagd-ui and flagd containers); flagd hot-reloads via its file watcher.
Discrepancy: the task's assumed flag names (paymentServiceFailure, cartServiceFailure) don't exist. Real keys are paymentFailure and cartFailure (no "Service" infix). kafkaQueueProblems and productCatalogFailure matched as assumed.

Mutation semantics differ per flag — must replicate flagd-ui's own storage.ex logic exactly:

  • Flags with a targeting.if block (currently only productCatalogFailure) → set targeting.if[1] (the "then" branch), leave defaultVariant untouched.
  • All other flags → set defaultVariant directly.
  • paymentFailure / cartFailure use percentage variants ("off", "100%", etc.), not booleans — use "100%" for deterministic failure.

F. Kafka → accounting propagation (group2)

fraud-detection (Kotlin) only logs to stdout — no queryable state, not host-reachable. accounting (.NET) writes to Postgres astronomy_db, schema accounting, table "order"(order_id TEXT PRIMARY KEY) (src/postgresql/init.sql:28-30). Neither service exposes a host port; Postgres itself uses ports: - ${POSTGRES_PORT} (single-value form), which Docker maps to a random ephemeral host port on every stack restart (currently 5673 → 5432) — not safe to hardcode.

Decision (confirmed with user): verify via docker exec astronomy-db psql -U astronomy_user -d astronomy_db -c "SELECT order_id FROM accounting.\"order\" WHERE order_id = '<id>'", shelled out from a test helper. Avoids the ephemeral-port problem entirely and needs no new npm dependency — only requires the Docker CLI on PATH.

G. Live-verified mutating mechanisms

Three assumptions were flagged by an independent review as unverified and risky enough to break Phase 2 silently. All three are now live-confirmed against the running stack (fully reversible, scoped to this local sandbox):

Flag write + propagation timing live-confirmed
Full round trip performed: GET /feature/api/read → mutated productCatalogFailure.targeting.if[1] to "on" → POST /feature/api/write → 200 → polled GET /api/products/OLJCESPC7Z and got 500 on the first poll (≤313ms) — propagation is near-instant. Confirmed a different product ID (66VCHSJNUP) stayed 200 throughout, proving the targeting scope. Reverted and diffed the full flag file against the pre-test baseline — byte-identical.
Valid credit card payload live-confirmed via source
src/payment/charge.js validates via simple-card-validator (Luhn + digit pattern), a brand allowlist (visa/mastercard only), and a future-expiry check. The demo's own frontend hardcodes a guaranteed-valid card at CheckoutForm.tsx:52-55: 4432-8015-6152-0454, cvv 672, exp 2030-01 — this exact literal will be reused in test payloads (documented as the demo's own known-good test card, not a real card).
OrderItem.cost is per-unit, not line-total
checkout/main.go:546-549 builds OrderItem.Cost from the converted unit price with no quantity multiplication; quantity is applied separately only when computing the order's internal grand total (not returned to the client). Group2's cross-check must multiply item.cost × item.quantity itself.
Guard against vacuous passes: every group3 test asserts a baseline 200 with the flag off before flipping it on, so a pre-existing broken payload can't make the "expect 500 after flag on" assertion pass for the wrong reason.
Whole-suite concurrency: /feature/api/write mutates one flag file shared by the entire running stack, not just group3 — if group3 ran on a separate worker while group1/group2 executed concurrently, flipping paymentFailure=100% would 500 group2's checkout mid-flight. Fix: workers: 1 for the whole playwright.config.ts project (nothing in this suite requires parallelism against one shared mutable backend).

group1-functional-api.spec.ts

Demonstrates direct per-service contract testing.

  • GET /api/products → 200, array of Product-shaped items
  • GET /api/products/OLJCESPC7Z → 200, correct shape
  • GET /api/products/{invalid} → 500 (documented as the actual, non-ideal contract)
  • POST /api/cart (add item) → 200, item present in returned cart
  • GET /api/cart → 200, hydrated item/product data
  • DELETE /api/cart (empty) → 204
  • POST /api/checkout with a valid cart, using the demo's own known-good test card → 200, orderId present, items priced (unit cost — × quantity for line totals)
  • Negative cases per corrected findings: empty-cart checkout (200, 0 items, shipping-only total) and negative-quantity add (200, accepted verbatim) — each documents the actual permissive behavior, not a guessed rejection

group2-integration-flow.spec.ts

Demonstrates cross-service integration testing.

  • Full flow: unique generated userId → add item to cart → checkout (same known-good test card) targeting a non-USD currency (e.g. EUR) → assert response items/shippingCost currencyCode matches, cross-check the converted item price against an independent GET /api/products/OLJCESPC7Z?currencyCode=EUR call (avoids hardcoding a conversion rate); multiply item.cost × item.quantity before any line-total comparison
    Two currency sources: items[].cost/shippingCost are driven by userCurrency in the checkout request body (passed to gRPC placeOrder); the ?currencyCode query param only re-hydrates items[].item.product.priceUsd. Set both to the same target currency.
  • Kafka propagation: after a successful checkout, poll accounting."order" via docker exec astronomy-db psql for the returned orderId (short retry loop, async consumption)

group3-fault-injection.spec.ts

Demonstrates resilience/fault-injection testing across service boundaries.

  • test.describe.configure({ mode: 'serial' }) for the whole file (belt-and-suspenders with the suite-wide workers: 1)
  • Reusable setFlag/resetFlag helper, built on the live-confirmed read→mutate→write round trip, with a short bounded poll (~2-3s) after write even though propagation measured ≤313ms live
  • Every test asserts a baseline 200 with the flag off first, then flips it on
  • paymentFailure (100%): baseline 200 → flag on → checkout → 500; reset in afterEach via try/finally
  • cartFailure (100%): baseline 200 on DELETE /api/cart → flag on → same call → 500; reset
  • productCatalogFailure (on): baseline 200 on GET /api/products/OLJCESPC7Z → flag on → 500, while a different product ID stays 200 the whole time; reset
  • kafkaQueueProblems (on): baseline 200 checkout → flag on → checkout still 200 (corrected resilience assertion); reset
  • All flag resets happen in afterEach unconditionally (try/finally), independent of test outcome
