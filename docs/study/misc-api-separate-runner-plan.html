Split misc-api onto its own GitHub Actions runner

Answering: could the misc-api Playwright project run on a separate machine in GitHub Actions?

Context

api.config.ts defines 9 Playwright projects: 8 GRA brand+region projects (pla-au, skx-au, drm-au, van-au, pla-nz, skx-nz, drm-nz, van-nz) running the shared gra-*.spec.ts suite, plus a 9th project, misc-api, which runs the two non-GRA specs (restful-booker.spec.ts, objects-crud.spec.ts — third-party public APIs, unrelated to the GRA staging environment).

Today, both .github/workflows/api-restful-tests.yml and .github/workflows/api-restful-tests-with-slack.yml run npm run test:api (all 9 projects) in a single job on a single ubuntu-latest runner, relying on Playwright's internal workers: 8 for intra-machine parallelism. misc-api shares that one VM with the GRA suite even though it targets a completely different backend with different flakiness characteristics.

Answer: yes — each GitHub Actions job already gets its own fresh runner VM by default, and jobs without a needs: dependency between them run in parallel on separate machines. The fix is to split the single api-tests job into two independent jobs.

Scope, per decision: this change targets api-restful-tests-with-slack.yml only. The plain api-restful-tests.yml and run-api-tests-nonparallel.sh stay untouched.

Resulting job topology

api-tests-gra

ubuntu-latest · runs 8 GRA projects · own monocart trend cache + CF Pages deploy

api-tests-misc

ubuntu-latest (separate VM) · runs misc-api only · own monocart report + artifact

notify-slack

needs: [api-tests-gra, api-tests-misc] · combines both results into one Slack message

Changes

1. package.json — two new project-scoped scripts

"test:api:gra": "playwright test --config=api.config.ts --project=pla-au --project=skx-au --project=drm-au --project=van-au --project=pla-nz --project=skx-nz --project=drm-nz --project=van-nz",
"test:api:misc": "playwright test --config=api.config.ts --project=misc-api",

Additive only — test:api (all 9 projects) is untouched and still backs the plain workflow.

2. .github/workflows/api-restful-tests-with-slack.yml — split into two parallel jobs

Mirrors the multi-job pattern already used in playwright-with-slack.yml (ui-test + api-test + test-report + notify-slack), scaled down to two peer jobs feeding one Slack step.
