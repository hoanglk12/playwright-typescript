4. Options Compared

Option A — Read-back from the live Cloudflare Pages deployment RECOMMENDED

Before generating the landing page, the CI job fetches the currently-deployed history.json from https://playwright-lighthouse-reports.pages.dev/history.json (treating a 404 as an empty first-run history), appends this run's record, and redeploys the updated file alongside the regenerated index.html. The generator renders inline-SVG sparklines — no charting library needed.

ProsCons
  • Literally "within your Reporting tool" — satisfies the ask exactly
  • Zero new npm dependencies
  • Zero new infrastructure or server-side code
  • Immune to GitHub Actions cache eviction
  • Matches this repo's established "plain Node.js, no new deps, static host" precedent
  • Single point of failure if the Cloudflare Pages project is ever deleted (mitigated with a 90-day artifact backup)
  • Generator must be extended to read per-run report JSON, not just the manifest

Option B — GitHub Actions cache (reuse monocart pattern) NOT RECOMMENDED

Store history.json in a branch-scoped actions/cache, following the same mechanism CLAUDE.md documents for monocart trend charts.

ProsCons
  • Reuses a mechanism already familiar in this repo
  • The existing monocart implementation uses a static cache key — actions/cache skips the post-job save on an exact-key match, so copied verbatim, history never actually grows
  • Fixing this needs a unique key per run + restore-keys prefix, adding complexity
  • GitHub evicts cache entries unused for >7 days — sits exactly on the boundary of the weekly-production cadence, risking silent history loss
  • History is invisible outside CI — doesn't live "within the Reporting tool" itself

Option C — LHCI Server mode (@lhci/server) NOT RECOMMENDED

Google's official Lighthouse CI server provides native trend graphs and build-to-build regression comparison out of the box.

ProsCons
  • Native trend/regression dashboard, maintained by Google
  • Richer comparison features than a hand-rolled sparkline
  • Requires standing up and hosting a persistent Node server + database (SQLite/MySQL/Postgres)
  • Replaces the current static dashboard rather than extending it — does not satisfy "within your Reporting tool"
  • Contradicts the static-host, zero-infra precedent this repo has followed so far
  • New hosting cost and operational burden for an incremental ask

5. Recommendation

Recommended: Option A — read-back & append history on the live Cloudflare Pages deployment
  • Fetch history.json from the live dashboard URL before each run (empty on first run / 404)
  • Extend generate-lhci-index.js to read CWV values from each representative run's report JSON
  • Append one record per {storefront, environment, formFactor}, trimmed to the last N runs (proposed: 26 ≈ 6 months weekly)
  • Render inline-SVG sparklines per storefront/metric below the existing snapshot table — no new dependencies
  • Redeploy history.json alongside index.html so the site is its own history store
  • Back up history.json as a 90-day GitHub Actions artifact in case the Pages project is ever lost
  • Never merge staging and production series — segment every line by environment and form factor

This dominates Option B on durability (no cache-eviction risk) and Option C on scope/cost (no new infrastructure), while being the only option that literally places the trend "within" the existing Reporting tool as the reviewer requested.

Rollback Plan

The trend feature is purely additive. To roll back: revert generate-lhci-index.js to the current snapshot-only version and remove the fetch-history and artifact-backup workflow steps. The live dashboard reverts to today's single-snapshot table on the next run. Deleting the deployed history.json is optional and harmless.

1

Confirm segmentation keys and retention window

Confirm with stakeholder: segment by {environment, formFactor, storefront}; retain last 26 runs per series (≈ 6 months at weekly cadence).

2

Extend scripts/generate-lhci-index.js

Accept an existing lighthouse-reports/history.json (default [] if absent). For each representative manifest entry, open its jsonPath report and extract numericValue for LCP / CLS / TBT / FCP. Append one record per storefront/form-factor with timestamp, trigger, commit, the 4 category scores, the CWV values, and an environment tag. Trim each series to the last N runs. Write history.json back out. Render an inline-SVG sparkline per storefront per metric below the existing table. Keep the script pure and network-free — the workflow does the fetching.

3

Add a history fetch step to lighthouse-ci.yml

Insert before the landing-page generation step:

- name: Fetch existing trend history
  if: always()
  run: |
    curl -f https://playwright-lighthouse-reports.pages.dev/history.json \
      -o lighthouse-reports/history.json \
      || echo '[]' > lighthouse-reports/history.json
  continue-on-error: true

Pass trigger/environment metadata to the generator via new env vars, reusing the existing GITHUB_SHA / GITHUB_REF_NAME pattern (e.g. add LHCI_ENVIRONMENT, LHCI_TRIGGER).
