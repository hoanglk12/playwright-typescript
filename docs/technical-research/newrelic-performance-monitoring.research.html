2. Summary

Hoang's comment is anchored on the tool-selection row, but its substance (dashboards, trend history, cross-brand coverage) maps to concerns raised later in the same review thread by others: comment 4064641033 ("could we turn this into a dashboard?"), comment 4113760262 ("include trend monitoring within your reporting tool"), and comment 4113793037 ("is New Relic RUM something we can currently set up/configure?").

The proposal itself already commits to New Relic in §3.7:

"INP monitored via New Relic Browser / CrUX field data for real-user validation alongside Lighthouse lab results"

This is not an either/or decision. Lighthouse CI is lab/synthetic testing run pre-deploy against staging. New Relic Browser is field/RUM monitoring of real production users, already instrumented on every brand. Hoang's idea is best read as "surface the New Relic data we already have to close the gaps the reviewer is asking about," not as a competing tool to replace Lighthouse.

5. Options Compared

Option A: Keep Lighthouse CI as the gate, add New Relic Browser as the field-data trend layer RECOMMENDED

Lighthouse CI stays exactly as configured (pre-deploy lab gate on staging, warn-only per §3.6). New Relic Browser, already instrumented on all 5 brands + TAF, becomes the source for week-on-week CWV trend monitoring, real INP validation, and cross-brand dashboard standardization.

ProsCons
  • Zero new CI runtime cost: RUM data is already flowing, no new instrumentation needed
  • Closes the exact gap the proposal itself flagged (INP cannot be measured in Lighthouse lab runs)
  • Directly answers three outstanding reviewer comments (dashboard, trend, RUM feasibility)
  • Preserves the working Lighthouse CI gate, artifacts, and Cloudflare dashboard
  • Dashboard/NRQL authoring work required across 5 NR accounts (manual, no IaC found in this repo)
  • Requires NR dashboard-edit access from whoever implements it
  • Data-retention window needs confirmation before publishing exact numbers

Option B: Replace Lighthouse CI with New Relic Synthetics scripted-browser monitors NOT RECOMMENDED

Configure NR Synthetics SCRIPT_BROWSER monitors to run Lighthouse-equivalent audits against staging on a schedule, replacing the GitHub Actions Lighthouse job entirely.

ProsCons
  • Single vendor for both synthetic and field monitoring
  • NR Synthetics has no native Lighthouse-equivalent a11y/SEO/best-practices scoring, so it would require rebuilding audit logic NR doesn't provide
  • Loses the existing Cloudflare Pages dashboard and assertion-based hard-gate model already working in this repo
  • Loses the PR-blocking artifact integration already in .github/workflows/lighthouse-ci.yml
  • Higher rebuild cost for a worse feature match, and it throws away a working POC

Option C: New Relic Browser only, drop Lighthouse entirely REJECTED

Rejected outright. Forfeits pre-deploy gating: NR RUM only sees traffic after deploy, so regressions ship before being caught. Forfeits accessibility/SEO/best-practices auditing entirely (NR has no equivalent). Forfeits the ability to test pages with no real traffic yet (new PDPs, pre-release pages). Does not meet the proposal's stated goal of blocking regressions before production (§3.2, §4 Phase 2).

6. Recommended Approach

Option A. The proposal already anticipated this combination in §3.7. Hoang's suggestion isn't a competing tool choice; it's the missing execution detail for a data source the document already commits to.

  • Standardize the "Customer experience quality foundation" dashboard pattern (currently DRM-only) across Skechers, Vans, Platypus, and TAF accounts
  • Replace the firstInputDelay widget with interactionToNextPaint (confirmed available and populated) in every brand's dashboard
  • Add a week-on-week trend view (TIMESERIES 1 week) to directly answer reviewer comment 4113760262
  • Confirm actual PageViewTiming data retention with the NR account admin before stating "months" in the proposal; state the observed ≈32-day window and its ambiguity instead
  • Update proposal §3.7 to make explicit that Lighthouse (lab, pre-deploy, gate) and New Relic Browser (field, post-deploy, trend + INP) are complementary, not alternatives
