Executed & verified Scope: local machine only CI impact: none Executed 2026-07-21

Per-machineTwo gotchas hit during the actual run — not in upstream release notes

1. Startup now requires a root .env file. lightrag-server ≥1.5.3 added a "multi-instance support" check that blocks or crashes startup if no .env exists in the working directory. Because .env* is gitignored, this file will not exist on a fresh machine even after pulling this repo — it must be recreated there too. Minimum content: LIGHTRAG_RUNTIME_TARGET=host. It does not interfere with Node's .env.{NODE_ENV} loading.

2. Unicode banner crash on Windows. The new splash-screen banner uses box-drawing characters that crash on a cp1252 console codepage with UnicodeEncodeError. This one can travel with git — the fix (set PYTHONIOENCODING=utf-8) is already applied inside scripts\start-rag.bat on this machine, but as an uncommitted change. It only reaches another machine once this repo change is committed and pushed. No action needed there once it lands, only mentioned so the cause is understood if seen elsewhere (e.g. running lightrag-server directly instead of via the script).

Neither fix has shipped yet. scripts/start-rag.bat, scripts/lightrag-requirements.txt, and this document itself are still uncommitted on this machine. Commit and push them before assuming another machine benefits from any of the above.

Why upgrade

The installed backend, lightrag_hku 1.5.2, is two patch releases behind the latest stable 1.5.4. Release notes for 1.5.3 and 1.5.4 document no changes to the REST or MCP surface this project depends on — query_document, insert_file, get_documents, delete_by_doc_ids, check_lightrag_health, get_pipeline_status are all unaffected. What changed is a set of security and reliability fixes: a guest-token authentication bypass was closed, X-API-Key enforcement was tightened, path-traversal in asset materialization was prevented, and a CORS wildcard-with-credentials conflict was resolved.

Confirmed0.0.0.0 bind is intentional

Since the wide bind is a deliberate choice rather than an oversight, this upgrade isn't being framed as a fix for the bind itself. It matters because .lightrag.env still has no LIGHTRAG_API_KEY or AUTH_ACCOUNTS set — so the server is unauthenticated on whatever network it's reachable from, and 1.5.4's fixes land squarely on that exposure class regardless of which interface it's bound to.

Separate findingmcp__lightrag__* has no backing server

The mcp__lightrag__* tools referenced in .claude/settings.json permissions, CLAUDE.md, the vault-updater agent, and the sync-vault skill aren't resolving to a registered MCP server — consistent with ToolSearch("lightrag") returning nothing. This is independent of the version bump and out of scope for this runbook, but it means the MCP-level validation step below is currently not runnable and is marked accordingly. The vault sync pipeline itself talks to the REST API directly (scripts/sync-vault-to-lightrag.mjs), so it isn't affected by this gap.

Upgrade steps

  1. Stop the server

    Confirm nothing is bound to port 9621 before touching the venv.

    netstat -ano | findstr :9621

  2. Back up the working directory

    File-based storage (nano_vectordb + networkx) means a directory copy is a complete, restorable snapshot.

    xcopy .lightrag .lightrag.bak-1.5.2 /E /I /H

  3. Activate the venv and record the current version

    Keeps a reference point for rollback.

    .lightrag-venv\Scripts\activate pip show lightrag-hku

  4. Install 1.5.4, pinned

    Stable only — do not install 1.5.5rc1.

    pip install "lightrag-hku==1.5.4"

  5. Commit a version pin

    .lightrag-venv and .lightrag.env are gitignored with no lockfile, so a fresh machine setup currently installs whatever's latest rather than a known version. Done on this machine, but still uncommitted — must be committed and pushed before another machine's fresh setup can install straight from this file (pip install -r scripts/lightrag-requirements.txt) instead of installing latest then upgrading.

    echo lightrag-hku==1.5.4 > scripts\lightrag-requirements.txt

  6. Create the root .env file

    Gitignored, so this does not travel with the repo — required on every machine, not just this one, or startup blocks/crashes on the 1.5.3+ multi-instance check. The check only requires the file to exist; LIGHTRAG_RUNTIME_TARGET is optional and any value other than host/compose/docker fails validation, so leaving it unset is also safe — an empty file passes too.

    echo LIGHTRAG_RUNTIME_TARGET=host > .env

  7. Restart and confirm clean startup

    Watch console output for startup errors before relying on the instance. The Unicode-banner fix is already baked into the script, so this should run clean with no extra flags.

    scripts\start-rag.bat

Rollback

If validation fails, revert in four steps — state is file-based, so the backup is a full restore:

  1. Stop lightrag-server
  2. Delete .lightrag, restore .lightrag.bak-1.5.2 in its place
  3. Inside .lightrag-venv, run pip install "lightrag-hku==1.5.2"
  4. Restart via scripts\start-rag.bat and re-check /health
