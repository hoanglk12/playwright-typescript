Multi-Page-Type Lighthouse Audits: PDP, PLP & Wishlist

Technical Research Report: Playwright TypeScript Framework
Research Only, Not Implemented CI/CD Change @lhci/cli ^0.15.0 Phased Rollout

1. Research Objective

Today's Lighthouse CI setup audits exactly one page type per storefront: the homepage. This report investigates (a) whether PDP (product detail), PLP (product listing), and Wishlist pages can be added to the audit, and (b) what the report landing page (scripts/generate-lhci-index.js) should look like once multiple page types exist per storefront, since the current layout is built around exactly one row per storefront with no concept of "page type."

2. Executive Summary

Feasible: yes, technically. LHCI's ci.collect.url already accepts an arbitrary URL array, and per-page-type assertion thresholds are natively supported via assertMatrix / matchingUrlPattern. But it cannot ship as a single change, for three independent reasons:

#BlockerDetail
1 Runtime blowout 8 → 32 URLs takes the run from ~48 min to ~192 min at 3 runs, ~320 min at the number_of_runs=5 dispatch option. The latter approaches GitHub's 6-hour hosted-runner ceiling and far exceeds today's timeout-minutes: 90. Fix: a GitHub Actions job matrix, not a timeout bump.
2 Data readiness Only homepage and wishlist have usable stable URLs today. pdpPath is a placeholder on all 8 storefronts. A bad slug on a Magento PWA SPA most likely renders a soft-404 (HTTP 200), producing plausible-but-wrong scores rather than failing loudly. PLP has no URL field at all.
3 Wishlist is degenerate for performance An anonymous LHCI crawl (no cookies/session) always hits the empty-state wishlist ("You have no items in your list."), confirmed on all 8 storefronts. Valid for a11y/BP/SEO, but misleading for the performance metric: the LCP element is empty-state text, not a product grid.
Recommended shape
  • Phased rollout: Wishlist → PLP → PDP
  • A shared lighthouse-urls.js module consumed by both LHCI configs and the generator
  • Option A (section-per-page-type) report layout
  • Per-page-type thresholds via assertMatrix
  • Bake a pageType key into the (not-yet-approved) trend history.json shape now, to avoid a later schema migration

Recommended lever: a GitHub Actions job matrix, not a timeout-minutes bump. Because numberOfRuns is global (no per-URL override), the only real concurrency lever is job-level. A 2-axis matrix (form-factor × page-type) yields up to 8 parallel jobs of ~24 min each, and every shard stays under today's budget while the feedback loop stays fast. A timeout bump to 240 min would technically work at 3 runs but degrades the feedback loop, burns more Actions minutes, and still breaks at number_of_runs=5.

Matrix consequence to plan for: splitting into N jobs breaks today's single-job assumption that the generator reads exactly one desktop + one mobile manifest.json. A dependent aggregation job must download all shard artifacts and concatenate their manifests into one array before generate/deploy.

Page typeData sourceStatusVerdict

Wishlist {baseUrl}wishlist (suffix on url) READY Easiest to wire; exclude from performance trend or hard-label "empty-state" (see below)
PLP new plpPath field (evergreen category URL) NEW DATA DEPENDENCY Must add field + verify stability; avoid promo/seasonal paths
PDP pdpPath field HARD BLOCKER Placeholder on all 8 sites; requires real, verified Magento slugs before this is meaningful

PDP silent-failure risk: a bad slug on a Magento PWA SPA most likely renders a soft-404 (HTTP 200 "product not found"). LHCI will not fail collect; it will produce plausible-looking scores for the wrong page and pollute trend data. This is more dangerous than a hard failure and must be framed as a blocking, silent-failure data-quality gate, not just a "nice to have."

PLP stability: use an evergreen womens/mens category URL, explicitly not the seasonal saleNavLabel targets (DM NZ's "BLACK FRIDAY" → /shop/sale is exactly the kind of rotating path that would 404/redirect mid-quarter).

Wishlist degeneracy (split verdict): an anonymous crawl always hits the empty state. This is degenerate for performance (empty-state text becomes the LCP element, not a product grid) but still valid for accessibility / best-practices / SEO (the empty-state markup and sign-in prompt are real page content). Recommendation: include wishlist for a11y/BP/SEO, but exclude it from the performance trend or hard-label it "empty-state" so it does not create a misleadingly-green performance line.

7. Labeling Mechanism (URL → {storefront, pageType})

Today's labelFromUrl() matches hostname only and cannot distinguish page types. Two approaches were considered:

ApproachVerdict
Path-regex heuristic in the generator (/wishlist → Wishlist; match configured pdpPath/plpPath per site → PDP/PLP; root → Homepage) Fragile: PDP/PLP paths are storefront-specific, not a shared pattern across brands
Shared URL module: a plain lighthouse-urls.js/.json exporting exact { url, storefront, pageType } tuples, consumed by both LHCI configs and the generator Recommended: kills URL duplication between configs, replaces the fragile heuristic with an exact lookup, and hands the pending trend feature its pageType value for free

Constraint: the LHCI CLI cannot import a .ts module, so this shared module must be plain .js/.json, a small curated projection of storefronts.ts data, not a direct import of it.

8. Assertion / Threshold Implications

A uniform performance minScore 0.9 / LCP ≤ 2500 ms is not realistic across all page types. PDPs carry heavy image galleries, and empty-state wishlist scores will be misleadingly high. LHCI's assertMatrix with matchingUrlPattern is the native mechanism to scope different thresholds to different URL patterns. Recommended structure:

  • Keep today's thresholds for homepage
  • Relax performance / LCP for PDP (heavier media) and PLP (product grids) via matchingUrlPattern-scoped blocks
  • Keep the accessibility error gate (minScore 0.9, image-alt, document-title, html-has-lang) uniform across all page types: it is a compliance gate, not a performance one, and must not be relaxed

9. Report Layout Options

Option A: Section-per-page-type RECOMMENDED

Keep today's exact one-table-per-form-factor structure, but nest an additional section level: Desktop → Homepage (8-row table) / PDP (8-row table) / PLP (8-row table) / Wishlist (8-row table), same again under Mobile.

ProsCons
  • Minimal deviation from the current renderTable/mobileSection pattern
  • Preserves the full 4-category table detail the accessibility error-gate depends on
  • Matches the "plain Node.js, no new deps" precedent exactly
  • Easiest to build incrementally: add one section per phase (Wishlist, then PLP, then PDP)
  • More vertical scrolling
  • Side-by-side page-type comparison requires scrolling between sections

Option B: Matrix table per form factor NOT RECOMMENDED

Rows = storefront, columns = page type (Home/PDP/PLP/Wishlist), each cell a compact score with a link to the full report.

ProsCons
  • Denser, more scannable for comparing page types side by side
  • Compresses to one score per cell, losing the 4-category detail the a11y gate surfaces
  • Requires a wholly new render function, not an extension of the existing one
  • Harder to read on mobile-width viewports

Option C: Tabs / JS toggle between page types NOT RECOMMENDED

A single-screen view with client-side tabs switching between page types.

ProsCons
  • Compact, single-screen
  • Introduces the first client-side JS into an otherwise fully static-generated page, a precedent shift worth flagging explicitly
  • Adds no data value over Option A

Note: the URL→{storefront, pageType} map (Section 7) is required by both A and B. It is not a differentiator between them. The real differentiator is rendering complexity and information density, where A wins on preserving the 4-category accessibility detail.

10. Recommendation

Recommended: phased rollout + shared URL module + Option A layout + assertMatrix thresholds
  • Ships value immediately without waiting on missing PDP data
  • Ties runtime risk and data-quality risk to concrete, phase-specific gates
  • The shared .js/.json URL module removes config duplication and fixes the duplicate-label defect
  • Hands the pending trend-monitoring feature its pageType value for free
  • Option A preserves the accessibility error-gate detail

Phase 1: Wishlist

No data dependency. Include for a11y/BP/SEO only; exclude or label performance as "empty-state." Re-measure real runtime before proceeding.

Phase 2: PLP

Add evergreen plpPath per storefront; verify stability against staging. Introduce the Actions matrix here if runtime demands it.

Phase 3: PDP GATED

Blocked until the team supplies real, verified, in-stock product slugs for all 8 storefronts. Add a collect-time URL health check to catch soft-404s.

Conditions: real PDP slugs verified live before Phase 3; PLP URLs confirmed evergreen (not promo); Actions matrix (or reduced numberOfRuns) in place before URL count crosses the ~16-URL / ~96-min line; pageType baked into history.json shape now if the trend feature ships first.

Not recommended if: the team cannot supply stable PDP slugs (skip Phase 3 entirely rather than audit soft-404s), or if there is appetite only for a timeout-minutes bump without the matrix (degrades the feedback loop and still breaks at number_of_runs=5).

Rollback Plan

Purely additive and staged. To roll back any phase: revert the URL module additions (drop the extra page-type tuples), revert generate-lhci-index.js to the prior renderer, and restore the single-job workflow. The dashboard reverts to the homepage-only table on the next run. No destructive changes to existing data.

Composability note: pending trend-monitoring research

This proposal and docs/technical-research/lighthouse-trend-monitoring.research.html overlap in scripts/generate-lhci-index.js and .github/workflows/lighthouse-ci.yml and should be sequenced deliberately. The trend research's history.json shape is keyed by {storefront, environment, formFactor}, with no pageType. Whichever ships second must not force a schema migration: bake a pageType key (default 'homepage') into the record shape now. Both features are additive and non-conflicting if the shared URL module lands first, since it supplies the pageType value both the layout and the trend series need.
