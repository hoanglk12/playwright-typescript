The Helper System

8 single-responsibility classes replacing direct page.* calls
⏳
WaitHelper this.waits
Page load, element visibility, network idle, custom poll conditions, spinner detection
🖱️
ElementHelper this.elements
Clicks, text input, getText, attributes, scroll, drag-and-drop, keyboard, dropdowns
🎨
StyleHelper this.style
Computed colours (→ hex), CSS properties, element dimensions, image natural size
🪟
FrameHelper this.frames
iframe operations via frameLocator() API — all iframe scoped interactions
📁
FileHelper this.files
File upload via input or drag-and-drop, multi-file handling, clear uploaded files
💾
StorageHelper this.storage
Cookies, localStorage, sessionStorage, clipboard read and write operations
🌐
NetworkHelper this.network
Route mocking, request interception, HAR recording, performance timing capture
📊
TableHelper this.tables
HTML table row/cell access, column sorting, pagination handling, bulk data extraction
🚫
Golden rule — never call page.locator() or page.click() directly inside a page class. Use this.elements.clickElement(), this.waits.waitForElement(), etc. All locators must be declared as private readonly class fields — never inline inside method bodies.
✅ Correct
class ProductPage extends BasePage {
  // 1. Locators hoisted to class fields
  private readonly addToCartBtn =
    this.page.getByRole('button', { name: 'Add to Cart' });
  private readonly priceSelector =
    '[data-testid="product-price"]';

  async addToCart(): Promise<void> {
    // 2. Delegate to helper
    await this.elements.clickLocator(this.addToCartBtn);
  }

  async getPrice(): Promise<string> {
    return this.elements.getText(this.priceSelector);
  }
}
❌ Banned
class ProductPage extends BasePage {
  async addToCart(): Promise<void> {
    // WRONG — direct page.click()
    await this.page.click('button:has-text("Add")');

    // WRONG — inline selector in helper call
    await this.elements.clickElement(
      '[data-testid="add-to-cart"]');

    // WRONG — structural CSS selector
    await this.page.locator(
      'div > section > ul > li:nth-child(2) > button')
      .click();
  }
}

Fixture-Based Dependency Injection

Page objects arrive via destructuring — no new calls, no shared state

src/config/base-test.ts wraps Playwright's test.extend() to register all page objects. Each fixture is lazily instantiated fresh per test. The use() boundary separates setup from teardown — code after await use(...) is guaranteed to run even if the test throws.

REST test example TypeScript 
import { apiTest as test } from
  '../../src/api/ApiTest';

test('TC_01 - Fetch product by ID',
  async ({ apiClientExt }) => {

  const res = await apiClientExt
    .getWithWrapper('/products/1');

  await res.assertStatus(200);
  await res.assertJsonPath('id', 1);
  await res.assertHasHeader('content-type');
  const name = await res.extract('name');
});
GraphQL test example TypeScript 
const GET_USER = `
  query GetUser($id: ID!) {
    user(id: $id) { id name email }
  }
`;

test('TC_02 - Query user',
  async ({ graphqlClient }) => {

  const gql = await graphqlClient
    .queryWrapped(GET_USER, { id: '1' });

  await gql.assertNoErrors(); // ← always first
  await gql.assertDataField(
    'user.id', '1');
});
⚠️
Never interpolate variables into GraphQL strings. Use the variables argument: queryWrapped(QUERY, { id }). String interpolation is an injection risk and breaks query caching. Also: always use errors?.length guards, never bare truthy if (errors) — an empty array [] is truthy.

07

Test Anatomy

The complete shape of a well-formed spec file
tests/frontsite/my-feature.spec.ts TypeScript 
// ① Always import from @config/base-test — never from @playwright/test
import { test, expect, softExpect } from '@config/base-test';
import { MyFeatureData } from '../../src/data/my-feature-data';
import { createTestLogger } from '../../src/utils/test-logger';

// ② Tags go in the describe() name — not in individual test() names
test.describe('My Feature @frontsite @regression', () => {

  // ③ TC_XX naming for new frontsite / admin / API tests
  test('TC_01 - Verify page title and navigation links are visible',
    async ({ myPage, softAssert }) => {

    // ④ Logger: one per test, same name as the test
    const logger = createTestLogger('TC_01 page title and nav links');

    // ⑤ Steps: numbered, action-oriented
    logger.step('Step 1 - Navigate to the feature page');
    await myPage.navigateToPage();

    logger.step('Step 2 - Verify title');
    const title = await myPage.getPageTitle();
    // Hard assertion — precondition for all subsequent steps
    expect(title).toContain(MyFeatureData.EXPECTED_TITLE);

    logger.step('Step 3 - Soft-verify independent UI elements');
    // Soft assertions accumulate — test continues past each failure
    softAssert.toBeTruthy(await myPage.isNavVisible(), 'Nav visible');
    softAssert.toBeTruthy(await myPage.isFooterVisible(), 'Footer visible');
  });

  test('TC_02 - Verify search returns results',
    async ({ myPage }) => {
    const logger = createTestLogger('TC_02 search results');

    logger.step('Step 1 - Perform search');
    await myPage.search(MyFeatureData.SEARCH_TERM);

    logger.step('Step 2 - Assert results visible');
    const count = await myPage.getResultCount();
    logger.verify('Results returned', '>0', count);
    expect(count).toBeGreaterThan(0);
  });
});

Test Data Management

Typed, never hardcoded — every data shape carries a named interface

Static expected values live as const objects annotated with a named interface. Dynamic / random data uses generator classes with explicit return types. Never hardcode strings in spec files.

Assertion Strategy

Two modes — pick the right tool for each check
🛑
Hard Assertions — expect
Fail immediately. Use for preconditions that guard subsequent steps. If a precondition fails, the rest of the test is meaningless — stop fast.

Also use for: Playwright locator assertions (expect(locator).toBeVisible()), expect.poll(), and single-check tests.
🔵
Soft Assertions — softAssert / softExpect
Continue past failure. Use for multiple independent checks on the same page state where all results matter.

softAssert (fixture) integrates with TestLogger — each call logs 🔵 [SOFT]. Do NOT call logger.verify() before it — that duplicates the log entry.
ScenarioUse
Multiple independent checks (title, count, visibility) on same stateSoft — see all failures at once
Per-item assertions inside a loopSoft — all items reported
Precondition that guards subsequent stepsHard — following steps are meaningless if this fails
Playwright locator assertions (expect(locator).toHaveText())Hard — SoftAssertHelper has no locator equivalent
expect.poll() with retry logicHard — poll has its own timeout/intervals
Single-assertion testHard — soft provides no benefit
