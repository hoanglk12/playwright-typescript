⛔ Top Gating Risk
Default headless Playwright Chromium receives HTTP 403 from Cloudflare. A spoofed Chrome user-agent returns HTTP 200 from a residential IP. Whether GitHub Actions datacenter IPs pass the same challenge is unverified and is the #1 blocker. Run a 1-test CI smoke before building the full suite.
⛔ Do Not Add KWH to storefronts[]
Every existing smoke spec iterates storefronts[]. Adding a Frontastic site to this Magento-shaped array will break all 8 GRA storefront specs. KWH must use its own isolated data module.
✅ Key Advantage
The site is richly instrumented with data-testid attributes (cart-icon-button, search-input, pdp-product-tag, etc.), making selectors far more stable than the CSS-class selectors used for GRA Magento storefronts.

4d. Helper Extensions

No new helper class is required. The existing composition pattern covers the interaction surface.

  • OverlayHelper may need additional dismiss selectors for KWH newsletter / cookie-consent overlays (additive, low risk — verify against a live render).
  • The UA override is a config-level concern, not a helper concern.

4e. CI/CD Changes

  • Add a new Chromium-only CI job in .github/workflows/playwright.yml (or a dedicated playwright-kwh.yml).
  • No new secrets required for unauthenticated browsing; add KWH test-account credentials later if auth flows are in scope.
  • Do not add KWH to the existing "Warm up staging storefronts" step — KWH is production CDN-fronted, not Azure cold-start.
⛔ CI Gate — Run This First
Before building the full suite, run a single 1-test Chromium smoke in CI with the UA override. If it returns HTTP 200, the integration is viable. If Cloudflare still challenges from datacenter IPs, the entire CI strategy needs revisiting (IP allowlisting or a non-Cloudflare lower environment).

5 Options Compared

Option A Not Viable

Add KWH to existing storefronts[] + reuse ecommerce POM

  • ✅ No new files
  • ❌ Breaks all 8 GRA specs — every smoke iterates storefronts[]
  • ❌ Magento selectors don't match Frontastic DOM
  • ❌ Interface fields (gender nav, size toggles, Qantas points) are meaningless for kitchenware
  • ❌ UA override leaks to all existing storefronts

Option B Recommended

Parallel KWH POM set + own data module + own spec suite + own Playwright project (Chromium-only)

  • ✅ Zero blast radius on existing 8 GRA specs
  • ✅ Stable data-testid selectors throughout
  • ✅ UA override isolated to the kitchenwarehouse project
  • ✅ Follows established composition-based POM pattern
  • ⚠️ More new files to create
  • ⚠️ Second commerce platform to maintain

Option C Future Only

Unified "ecommerce" abstraction over Magento + Frontastic behind a platform flag

  • ✅ One conceptual model
  • ❌ High complexity — leaky abstraction over two very different platforms
  • ❌ Risky refactor of a working suite
  • ⚠️ Only justified if 5+ Frontastic sites are planned

9 Open Questions

  • Is there a KWH staging or non-production environment? All probing was done against production. Automating cart/checkout against production risks creating real orders.
  • Will Cloudflare challenge GitHub Actions datacenter IPs even with the UA override? This is the single most important unknown — run a throwaway CI smoke immediately to find out.
  • What is the auth mechanism? /login returns 404 and /account redirects to home. Is auth (login/register/order history) in scope for the initial suite?
  • Chromium only, or also Firefox? Recommendation is Chromium-only for KWH (avoids Firefox Juggler teardown workaround and a second UA fingerprint to manage).
  • Separate CI workflow (playwright-kwh.yml) or extend existing playwright.yml? Separate is cleaner; extending avoids workflow proliferation.
