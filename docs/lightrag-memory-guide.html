🧠 What is a Knowledge Graph?

A knowledge graph is a data structure that represents information as a network of entities (nodes) connected by relationships (edges). Unlike a flat list of facts or a traditional vector search index, a knowledge graph captures how things relate to each other, allowing retrieval that follows conceptual links rather than just similarity scores.

🔵

Entities (Nodes)

Named concepts: BasePage, GRA API, Ecommerce PDP, Firefox teardown. Extracted automatically from your notes by an LLM.

🔗

Relationships (Edges)

Typed connections: BasePage OWNS ElementHelper, GRA API HAS 4 brands, EcommerceCartOverlayPage REQUIRES Firefox workaround.

🌐

Dual Retrieval

Local: entity neighborhood (what is directly related to this concept).
Global: theme-level reasoning across the entire graph.

Traditional RAG (Retrieval-Augmented Generation) splits documents into chunks and finds the nearest chunks by vector similarity. This works well for isolated facts but misses connections — it cannot answer "what are all the constraints that affect the Firefox ecommerce tests?" because the answer lives across six different notes.

A knowledge graph solves this by pre-computing the connections at ingest time, so queries can traverse the graph and retrieve semantically related context from multiple notes in a single hop.

✨ Why LightRAG + Obsidian?

The project stores its Claude Code memory in Obsidian — a markdown vault with wikilinks and a graph view. The vault at memory-vault/ is already structured with typed notes (user, feedback, project, reference) and wikilinks between them. LightRAG is the semantic layer on top.

Why not plain Grep?

Claude already uses Grep over the vault for keyword search. This covers 90% of lookups. LightRAG adds value when:

  • → The question spans multiple notes and requires synthesising relationships
  • → The keyword is not the exact word in the note (semantic similarity)
  • → You want a summary of all context related to a concept (e.g. "all GRA API quirks")
  • → You add new notes via Jira/Confluence and want them immediately searchable

⚙️ Configuration (.lightrag.env)

# ─── LLM (entity + relationship extraction) ───────────────────────────────
LLM_BINDING=openai
LLM_MODEL=gpt-4o-mini           # cheaper; gpt-4o for best quality
LLM_BINDING_HOST=https://direct.shopaikey.com/v1
LLM_BINDING_API_KEY=sk-your-api-key-here

# ─── Embeddings ────────────────────────────────────────────────────────────
EMBEDDING_BINDING=openai
EMBEDDING_MODEL=text-embedding-3-large
EMBEDDING_BINDING_HOST=https://direct.shopaikey.com/v1
EMBEDDING_BINDING_API_KEY=sk-your-api-key-here
EMBEDDING_DIM=3072               # text-embedding-3-large output dim

# ─── Paths ─────────────────────────────────────────────────────────────────

Never commit this file. The .gitignore already excludes .lightrag.env. If an API key is accidentally committed, regenerate it immediately — the key is compromised once it appears in git history.

🤖 Model Selection

LLM — entity extraction

The LLM is called during ingestion to extract entities and relationships from each document chunk. It is also called at query time to synthesise the final answer. Two sensible choices:

ModelInput costOutput costQualityRecommendation
gpt-4o $2.50 / 1M $10.00 / 1M Best Use for large vaults or high-value queries
gpt-4o-mini $0.15 / 1M $0.60 / 1M Good Default — 15× cheaper, sufficient for short notes

At the current vault size (~17 notes, ~13K tokens), even a full re-ingest with gpt-4o costs under $0.10. The model choice matters more for query-time cost if Claude queries LightRAG frequently. gpt-4o-mini is the right default.

Embeddings — vector index

The embedding model converts text chunks and extracted entities into dense vectors for similarity search. It is called at ingest time only (not at query time for most operations).

ModelCostDimensionsNotes
text-embedding-3-large $0.13 / 1M 3072 Best retrieval quality. Set EMBEDDING_DIM=3072.
text-embedding-3-small $0.02 / 1M 1536 6× cheaper. Set EMBEDDING_DIM=1536 if you switch.

If you change the embedding model after initial ingestion, you must delete .lightrag/ and re-ingest the entire vault. The vector dimensions are fixed at first ingest — a mismatch causes silent retrieval failures.

🚀 Starting the Server

The server is started by scripts/start-rag.bat. Run it in a dedicated terminal that stays open. It reads .lightrag.env by parsing each line with SET, activates the venv, and starts the HTTP server. 

@echo off
cd /d "%~dp0.."
echo Starting LightRAG server...
echo Vault: memory-vault/20-memory
echo Server: http://localhost:9621
echo Swagger: http://localhost:9621/docs

REM Load .lightrag.env by SET-ing each variable
REM (lightrag-server reads its own .env from cwd, which would conflict
REM  with the project-root Node .env files for Playwright)
for /f "usebackq eol=# tokens=1,* delims==" %%A in (".lightrag.env") do (
    if not "%%A"=="" if not "%%B"=="" set "%%A=%%B"
)

call .lightrag-venv\Scripts\activate
lightrag-server ^
    --host 0.0.0.0 ^
    --port 9621 ^
    --working-dir .lightrag ^
    --input-dir memory-vault/20-memory ^
    --llm-binding openai ^
    --embedding-binding openaistart-rag.bat

Verify the server is healthy

curl http://localhost:9621/healthPowerShell

Expected response:

{
  "status": "healthy",
  "working_dir": ".lightrag",
  "input_dir": "memory-vault/20-memory",
  "llm_model": "gpt-4o-mini",
  "embedding_model": "text-embedding-3-large"
}JSON

The Swagger UI is also available at http://localhost:9621/docs for manual testing.

The server must be running before the MCP bridge can connect. Start start-rag.bat first, verify health, then start Claude Code. If you restart Claude Code without restarting the server, the MCP bridge reconnects automatically.

📥 Ingesting the Vault

The LightRAG /documents/scan endpoint ignores .md files (it only scans known document types like PDF, DOCX). The correct ingestion path is POST /documents/text — one API call per file, sending the raw markdown content.

First-time ingest

Run the sync script (Node.js) which handles both insert and update logic:

node scripts/sync-vault-to-lightrag.mjsPowerShell

If the server is not running, the script detects the connection error and exits silently (non-fatal). Expected output for a fresh ingest: 

[sync-vault-to-lightrag] LightRAG healthy — syncing...
  Inserted: user_profile.md
  Inserted: feedback_preferences.md
  Inserted: ecommerce-pdp-page-gotchas.md
  Inserted: project_context.md
  Inserted: project_architecture.md
  Inserted: gra-api-testing.md
  ...
[sync-vault-to-lightrag] Done — 14 new, 0 updated, 0 unchangedstdout

What actually happened — the first successful bulk ingest

Before sync-vault-to-lightrag.mjs was written, the original approach was a PowerShell script (scripts/ingest-vault.ps1) that called POST /documents/scan. That failed silently (found 0 files — see Error 5 below). The replacement script was ingest-vault.ps1 rewritten to loop over each .md file and call POST /documents/text.

However, running .\scripts\ingest-vault.ps1 in the terminal opened the file in Notepad instead of executing it (Windows file-association block — see Error 6). The workaround was to run the equivalent logic as a PowerShell inline command pasted directly into the terminal: 

# Inline bulk ingest — used when .ps1 execution was blocked
# Run from the project root with LightRAG server already running
Get-ChildItem -Path "memory-vault/20-memory" -Recurse -Filter "*.md" |
  Where-Object { $_.DirectoryName -notlike "*__parsed__*" } |
  ForEach-Object {
    $content = Get-Content $_.FullName -Raw
    $body = @{ text = $content; file_source = $_.Name } | ConvertTo-Json -Depth 2
    Invoke-RestMethod -Uri "http://localhost:9621/documents/text" `
      -Method POST -ContentType "application/json" -Body $body
    Write-Host "Inserted: $($_.Name)"
  }PowerShell

Each file triggers an async LightRAG pipeline. The server processes them in batches — you will see log lines like INFO: Processing chunk 1/3 for user_profile.md in the start-rag.bat terminal window. When all files are through, the status endpoint confirms: 

curl http://localhost:9621/documents
# Look for "status": "processed" for each file_path
# All files showing "processed" = ingest complete ("all files chunked completed")PowerShell

The Node.js script (sync-vault-to-lightrag.mjs) supersedes this inline command — it adds diff detection (skips unchanged files), handles deletes before re-inserts, and is non-fatal when the server is down. Use the script for all normal operations; the inline command is documented here as a break-glass fallback.

Verifying ingest completed

LightRAG processes documents asynchronously. After inserting files, poll the pipeline status to confirm all chunks are processed before querying: 

# Check pipeline status
curl http://localhost:9621/documents/pipeline_status

# Check which docs are indexed
curl http://localhost:9621/documentsPowerShell

Expected successful state in /documents response:

{
  "statuses": {
    "processed": [
      { "id": "...", "file_path": "user_profile.md", "content_length": 842 },
      { "id": "...", "file_path": "gra-api-testing.md", "content_length": 3210 },
      // ... all 14+ notes
    ],
    "pending": [],
    "failed": []
  }
}JSON

If any files appear in "failed", the most common cause is the LLM API returning an error during entity extraction (rate limit, auth failure, or model unavailable). Check the .lightrag/lightrag.log file for the exact error, fix the .lightrag.env credentials, and re-insert the failed files.

Ongoing sync (automatic)

The Stop hook in .claude/settings.json runs sync-vault-to-lightrag.mjs automatically at the end of every Claude Code session. The script is idempotent — unchanged files are detected by content length and skipped.

How the sync script works

// 1. Health check — exits silently if server is down
const health = await fetch(`${LIGHTRAG_URL}/health`);

// 2. Get list of already-indexed docs from LightRAG
const docs = await fetch(`${LIGHTRAG_URL}/documents`);
// Build map: filename → { id, contentLength }

// 3. Walk memory-vault/20-memory/ recursively (skip __parsed__/)
// 4. For each .md file:
//    - If content length unchanged → skip
//    - If changed → DELETE old doc, then POST /documents/text
//    - If new → POST /documents/textJavaScript

Never call DELETE /documents (no body) — this wipes the entire document store. The correct per-document delete endpoint is DELETE /documents/delete_document with body {"doc_ids": ["id"]}.

🐛 Errors Encountered & Fixes

All errors hit during the initial setup on Windows 11, with the exact fix applied.

Error 1 — --env-file flag not supported by lightrag-server

Symptom: Running lightrag-server --env-file .lightrag.env failed immediately with an unrecognised argument error.

Fix

lightrag-server does not support --env-file. The start-rag.bat script instead parses .lightrag.env line-by-line using a for /f loop and calls SET for each variable before invoking the server: 

for /f "usebackq eol=# tokens=1,* delims==" %%A in (".lightrag.env") do (
    if not "%%A"=="" if not "%%B"=="" set "%%A=%%B"
)bat
Error 2 — Wrong MCP entry point / command

Symptom: The MCP server entry in .mcp.json used "command": "lightrag_mcp" which is not a valid executable name.

Fix

Use uvx with --from lightrag-mcp to run the correct executable:

{
  "command": "uvx",
  "args": ["--from", "lightrag-mcp", "lightrag-mcp", "--host", "localhost", "--port", "9621"]
}JSON
Error 3 — Wrong environment variable names (silent auth failure)

Symptom: Server started successfully but all LLM/embedding API calls returned 401. No error at startup — the wrong variable names were silently ignored.

Fix

LightRAG v1.5.x requires the _BINDING_ infix in key names:

Wrong ❌Correct ✓
LLM_API_KEYLLM_BINDING_API_KEY
EMBEDDING_API_KEYEMBEDDING_BINDING_API_KEY
Error 4 — Port 9621 already in use (second server instance)

Symptom: Running start-rag.bat a second time failed with OSError: [Errno 98] Address already in use.

Fix

The first instance was still running and healthy. Close the duplicate terminal. Check which process owns port 9621: 

netstat -ano | findstr :9621PowerShell

Kill by PID if needed: taskkill /PID <pid> /F

Error 5 — /documents/scan finds 0 files

Symptom: POST /documents/scan returned {"status":"success","found":0}. All 14 vault notes were present but not indexed.

Root cause

LightRAG's file scanner only recognises extensions like .pdf, .docx, .txt. It does not process .md files via scan.

Fix

Use POST /documents/text with the raw markdown content as the body. The sync-vault-to-lightrag.mjs script implements this correctly — it reads each .md file and POSTs its content individually. 

await fetch(`${LIGHTRAG_URL}/documents/text`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ text: fileContent, file_source: filename }),
});JavaScript
Error 6 — PowerShell script opens in text editor instead of running

Symptom: Double-clicking or typing .\scripts\ingest-vault.ps1 opened the file in Notepad / VS Code instead of executing it.

Fix

Windows associates .ps1 files with text editors by default as a security measure. Run scripts explicitly via the PowerShell executable, or use the Node.js alternative:

# Explicit invocation (always works)
PowerShell -ExecutionPolicy Bypass -File scripts\ingest-vault.ps1

# Or use the Node.js sync script instead (preferred)
node scripts/sync-vault-to-lightrag.mjsPowerShell
Error 7 — TOKEN_SECRET / .env startup warnings

Symptom: Server printed WARNING: TOKEN_SECRET not set and Warning: Startup directory must contain .env file at startup.

Root cause & fix

These are non-fatal warnings from LightRAG's auth middleware (JWT token secret) and its own .env file reader. Since the project already has a Node .env.* file for Playwright, these are normal. The server operates correctly despite these warnings. To suppress TOKEN_SECRET: add TOKEN_SECRET=any-random-string to .lightrag.env.

🎯 Benefits of This Approach

🔗

Graph-aware retrieval

Questions that span multiple notes — "what affects Firefox ecommerce tests?", "list all GRA API quirks" — get synthesised answers by traversing entity relationships, not just keyword matches.

🔍

Semantic search over vocabulary drift

A note about "Juggler protocol hang" is retrieved by a query about "Firefox teardown issue" even though none of those exact words appear together.

🔄

Automatic sync

The Stop hook chain means Claude's memory is always propagated to Obsidian and LightRAG without any manual intervention. Memory is durable across machines.

📚

Jira/Confluence integration

The vault-updater agent pulls real project knowledge from Jira/Confluence into the graph. External team decisions and documentation become queryable context.

🛡️

Non-intrusive to existing workflow

LightRAG is optional — if the server is down, sync scripts exit silently and Grep-over-vault continues to work. No single point of failure.

💰

Cost-effective

At 17 notes (~13K tokens), a full re-ingest with gpt-4o-mini costs under $0.01. Ongoing sync only processes changed files. Embeddings are computed once.

Prompt instruction — where to put it

The rule lives in memory-vault/.claude/CLAUDE.md under LightRAG query rule. Claude Code auto-loads all CLAUDE.md files in the project tree, so this instruction is active in every session without any manual step. 

**LightRAG query rule:** when the question spans multiple notes, requires relationship
reasoning ("what are all constraints for X?", "list everything that affects Y"), or Grep
returns no relevant results — call `mcp__lightrag__query` with the question as the query
string and `mode: "hybrid"`. Check `mcp__lightrag__check_lightrag_health` first; if the
server is not running, fall back to Grep.CLAUDE.md instruction
