What Makes a Test "Integration" on GRA

Integration tests verify that two or more system layers communicate correctly. On GRA, there are four boundaries worth testing.

Boundary A

Frontend → GraphQL → Magento

User action fires a GraphQL mutation or query. The backend processes it, returns structured data. The frontend re-renders from that data. Verifying the full round-trip catches serialization bugs, missing fields, and schema drift.

Boundary B

GraphQL Response → UI State

After a mutation (e.g. addProductsToCart), the mini-cart badge, overlay total, and item list must all reflect the server response — not a locally cached guess. Tests here catch React state hydration bugs.

Boundary C

UI Action → Analytics (dataLayer)

Every key action must fire the correct window.adobeDataLayer event with the right product/page data. Tests here catch GTM tag breaks that silently kill revenue reporting.

Boundary D

Cross-Storefront Localization

The same page object must render AUD on AU sites and NZD on NZ sites. Qantas Points must appear only on AU. Tests here catch config drift when GRA deploys to a new region.

ℹ
Not integration: clicking a nav link and asserting URL change (smoke). Integration requires verifying that two separate layers agreed on the same data — browser state matched server state, or UI event matched analytics payload.

P1 Critical Scenarios

These 6 scenarios must pass on every staging deploy. They protect the primary revenue path.

INT-001
Add to Cart — GraphQL Mutation ↔ UI ↔ Analytics
P1 Critical All 8 sites 3 Boundaries ★★★★★ ROI

The highest-value integration test. Validates that addProductsToCart mutation fires, Magento processes it, the mini-cart badge increments, the overlay total matches the PDP price, and an analytics event lands in adobeDataLayer.

Boundary A: GraphQL mutation → Magento cart
Boundary B: Magento response → UI state
Boundary C: ATC action → Analytics event

Test Steps

  • 1
    Block noise routes (TrueFit, FullStory, Facebook, Taboola)
  • 2
    Intercept addProductsToCart GraphQL request
  • 3
    Navigate → PLP → PDP (preferMens where required)
  • 4
    Capture productPrice BEFORE selectSize()
  • 5
    Dismiss Vans popup if present (#popup-close)
  • 6
    Select first available size → addToCart()

Assertions

  • ✓ Intercepted mutation response has cart.items.length > 0
  • ✓ Cart badge = initialCount + 1
  • ✓ Mini-cart overlay total ≈ PDP price (±0.01)
  • ✓ add_to_cart event in adobeDataLayer
  • ✓ Event product name matches PDP product name
// Intercept the GraphQL mutation before navigation
let mutationPayload: Record<string, unknown> | null = null;
await page.route('**/graphql?**operationName=addProductsToCart**', async route => {
  const response = await route.fetch();
  mutationPayload = await response.json();
  await route.fulfill({ response });   // pass through — don't mock
});

// After addToCart() — verify all 3 boundaries
const cartItems = (mutationPayload as any)?.data?.addProductsToCart?.cart?.items;
softAssert.toBeTruthy(cartItems?.length > 0, 'GraphQL cart has items');

const newCount = await pdpPage.getMiniCartCount();
softAssert.toBe(newCount, initialCount + 1, 'Cart badge incremented');

const atcEvent = await page.evaluate(() =>
  (window as any).adobeDataLayer?.getState?.() ??
  (window as any).dataLayer?.find((e: any) => e?.event === 'add_to_cart')
);
softAssert.toBeTruthy(!!atcEvent, 'add_to_cart dataLayer event fired');
⚠
Vans AU: Bloomreach popup (#popup.popup.visible, z-index 200) blocks cart icon click. Dismiss via #popup-close before any Vans interaction.
Capture price BEFORE selectSize() — Vans AU drops the ATC button on size selection (race condition).
INT-002
Full Revenue Path — Search → Attraqt → PDP → Add to Cart
P1 Critical All 8 sites 2 Boundaries ★★★★★ ROI

Tests the complete top-of-funnel revenue path. Verifies that Attraqt-powered search returns relevant results, clicking a search result lands on the correct PDP (URL/product name match), and the ATC button is enabled and functional — all in a single session.

Boundary B: Search result click → correct PDP loaded (URL + title consistent)

INT-006
GraphQL Error Fallback — Network Failure Handling
P1 Critical AU sites × 4 1 Boundary ★★★★☆ ROI

Deliberately fails specific GraphQL operations using Playwright route mocking to verify the storefront degrades gracefully — showing an error state or fallback UI rather than a blank page or JS crash. This catches missing error boundaries in the PWA Studio React app.

Boundary A: GraphQL 500 error → UI error state (not blank/crashed page)

Scenarios to Fail

  • A
    GetCmsPage → 500 → homepage still renders header/nav
  • B
    productSearch → empty results → "No results" message shown
  • C
    addProductsToCart → GraphQL error → toast or inline error shown
  • D
    generateCustomerToken → invalid credentials error → login error shown in modal

Assertions

  • ✓ Page does not show a blank white screen
  • ✓ Header and navigation remain functional
  • ✓ Error message or fallback UI visible to user
  • ✓ No uncaught React "Cannot read properties of undefined" in console
// Simulate GraphQL failure for a specific operation
await page.route('**/graphql?**operationName=addProductsToCart**', async route => {
  await route.fulfill({
    status: 200,   // GraphQL returns 200 even for errors
    contentType: 'application/json',
    body: JSON.stringify({
      errors: [{ message: 'Cart not found', extensions: { category: 'graphql-no-such-entity' } }],
      data: { addProductsToCart: null }
    })
  });
});

// Expect an error toast or inline message — not a crash
const errorVisible = await page.locator('[class*="toast" i], [class*="error" i], [role="alert"]')
  .isVisible({ timeout: 5000 }).catch(() => false);
expect(errorVisible).toBe(true);

INT-010
Spend & Save — Cart Threshold → Discount Ladder (Skechers NZ)
P2 High Skechers NZ only 2 Boundaries ★★★☆☆ ROI

Skechers NZ has a tiered "Spend & Save" promotion: $110 unlocks 15% off, $170 unlocks 20% off, $200 unlocks 25% off. This integration test seeds carts at each threshold via REST API, then verifies the correct discount tier is applied in the GraphQL cart response and displayed in the mini-cart UI.

Boundary A: Cart total at threshold → Magento rule engine → discount applied in customerCart response
Boundary B: GraphQL cart discount → UI shows correct discount tier and % off message
// Parameterised over 3 thresholds
const tiers = [
  { minCart: 110, expectedDiscount: 15, label: 'Spend $110, Save 15%' },
  { minCart: 170, expectedDiscount: 20, label: 'Spend $170, Save 20%' },
  { minCart: 200, expectedDiscount: 25, label: 'Spend $200, Save 25%' },
];

for (const tier of tiers) {
  // Seed cart via REST to exactly meet threshold
  await apiClient.seedCartToAmount(guestCartId, tier.minCart);

  // Open mini cart and verify discount is shown
  await cartOverlay.waitForOverlayVisible();
  const discountText = await cartOverlay.getDiscountMessage();
  softAssert.toContain(discountText, `${tier.expectedDiscount}%`, tier.label);
}

Test Setup Patterns

Reusable setup blocks that should appear in beforeEach for all integration tests.

Route Noise Blocking

Block these in beforeEach to reduce flakiness from slow third-party calls.

test.beforeEach(async ({ page }) => {
  for (const pattern of [
    '**/truefitcorp.com/**',    // fails on staging
    '**/fullstory.com/**',       // session recording
    '**/facebook.com/tr**',      // FB pixel
    '**/taboola.com/**',         // ads
    '**/paypal.com/**',          // widget pre-load
    '**/analytics.google.com/**',// GA4
    '**/demdex.net/**',          // Adobe DMP
  ]) {
    await page.route(pattern, r => r.abort());
  }
});

Vans Popup Guard

Must run before any click interaction on Vans AU/NZ. The popup is session-based — defensive check always.

async function dismissVansPopup(page: Page) {
  await page.locator('#popup.popup.visible')
    .waitFor({ state: 'visible', timeout: 3000 })
    .then(async () => {
      await page.locator('#popup-close').click();
      await page.locator('#popup.popup.visible')
        .waitFor({ state: 'hidden', timeout: 5000 });
    })
    .catch(() => {});  // Not shown — continue
}

// In test:
if (site.name.toLowerCase().includes('vans')) {
  await dismissVansPopup(page);
}

GraphQL Mutation Interceptor

Capture any GraphQL operation by name for assertion without mocking.

async function interceptGql(page: Page, opName: string) {
  let payload: unknown = null;
  await page.route(
    `**/graphql?**operationName=${opName}**`,
    async route => {
      const res = await route.fetch();
      payload = await res.json();
      await route.fulfill({ response: res });
    }
  );
  return { get: () => payload };
}

// Usage:
const mutation = await interceptGql(page, 'addProductsToCart');
await pdpPage.addToCart();
const items = (mutation.get() as any)?.data?.addProductsToCart?.cart?.items;

REST API Cart Seeding

Seed cart before checkout/promo tests — faster than UI ATC flow.

// POST /rest/V1/guest-carts → cartId
// POST /rest/V1/guest-carts/{cartId}/items
async function seedGuestCart(
  apiClient: ApiClient,
  baseUrl: string,
  sku: string,
  qty = 1
): Promise<string> {
  const { data: cartId } = await apiClient
    .post(`${baseUrl}/rest/V1/guest-carts`, {});
  await apiClient.post(
    `${baseUrl}/rest/V1/guest-carts/${cartId}/items`,
    { cartItem: { sku, qty, quote_id: cartId } }
  );
  return cartId;
}
