1
Composition over Inheritance
The central architectural choice of the whole framework
What it is

BasePage owns 8 helper instances as class fields. Subclasses extend BasePage but interact with the page through those helpers — never by calling page.locator() or page.click() directly. Each helper has one responsibility.

Why it matters

Inheritance hierarchies grow brittle. A change to a parent method breaks all subclasses. With composition, a scroll bug in ElementHelper is isolated — it doesn't affect WaitHelper or StyleHelper.

Interview talking point
Instead of a deep inheritance chain, BasePage composes 8 single-responsibility helpers. EcommercePDPPage extends BasePage, but when it needs to wait for an element, it calls this.waits.waitForCustomCondition(), not page.waitForSelector() directly. Each helper owns one concern — changing wait logic in WaitHelper doesn't touch element interactions.
Framework code
src/pages/base-page.ts — constructor, lines 41–51
BasePage — composition of 8 helpers in constructor
export abstract class BasePage {
  protected page: Page;

  // ── Helpers (use these in subclasses for new code) ──────────
  protected readonly waits: WaitHelper;
  protected readonly elements: ElementHelper;
  protected readonly style: StyleHelper;
  protected readonly frames: FrameHelper;
  protected readonly files: FileHelper;
  protected readonly storage: StorageHelper;
  protected readonly network: NetworkHelper;
  protected readonly tables: TableHelper;

  constructor(page: Page) {
    this.page = page;
    this.waits = new WaitHelper(page);
    // ElementHelper depends on WaitHelper — dependency injected
    this.elements = new ElementHelper(page, this.waits);
    this.style = new StyleHelper(page, this.waits);
    this.frames = new FrameHelper(page);
    this.files = new FileHelper(page, this.waits);
    this.storage = new StorageHelper(page);
    this.network = new NetworkHelper(page);
    // TableHelper composes both WaitHelper and ElementHelper
    this.tables = new TableHelper(page, this.waits, this.elements);
  }
}

export class EcommercePDPPage extends BasePage {
  // Locators declared as private readonly class fields (never inlined in methods)
  private readonly productNameHeading = this.page.getByRole('heading', { level: 1 }).first();
  private readonly addToCartBtnLocator = this.page.locator('button', { hasText: /add to (cart|bag)/i });
  private readonly PDP_URL_PATTERN = /(\/product\/|\/p\/|\/pdp\/|\.html)/i;

  constructor(page: Page) { super(page); }

  async waitForPdpLoad(): Promise {
    // Uses this.waits (WaitHelper), not page.waitForURL()
    await this.waits.waitForUrlMatches(this.PDP_URL_PATTERN, TIMEOUTS.PAGE_LOAD_SLOW);
    await this.productNameHeading.waitFor({ state: 'visible', timeout: TIMEOUTS.PAGE_LOAD });
  }

  async getProductName(): Promise {
    return (await this.productNameHeading.textContent())?.trim() ?? '';
  }
}

Facade Pattern
Stable public surface over a reorganised internal layer
What it is

BasePage keeps ~50 one-line delegation methods (clickElement, waitForElement, getText, …) that forward to the matching helper. Tests written before the helper refactor still compile and run without changes.

Why it matters

The internal structure changed (helpers introduced), but the external interface didn't. Zero call-site changes were needed. This is the core promise of the Facade: a stable front door that hides internal reorganisation.

Interview talking point
When we introduced the helper layer, we kept BasePage's existing public methods as thin delegations — clickElement() just calls this.elements.clickElement(). Tests written before the refactor required zero changes. That's the Facade pattern: a stable, simplified interface that hides the internal restructuring.

Strategy Pattern
Interchangeable algorithm selected at construction time
What it is

ApiClient uses an AuthType enum to select which authentication strategy to apply during init(). The calling test just declares an auth type and credentials — it doesn't know which headers get added.

Why it matters

Adding a new auth scheme requires only a new AuthType value and a new case — the interface used by tests stays unchanged.

Interview talking point
AuthType is an enum that drives a strategy switch in ApiClient.init(). The test just says AuthType.BEARER and passes a token. The header construction algorithm is swapped at construction time. If we add OAuth2, we add one enum value and one case — no test changes needed.
Framework code
src/api/ApiClient.ts — AuthType enum + init() strategy switch
ApiClient — strategy selection via AuthType enum
export enum AuthType {
  NONE    = 'none',
  BASIC   = 'basic',
  BEARER  = 'bearer',
  API_KEY = 'api_key',
  CUSTOM  = 'custom',
}

export class ApiClient {
  constructor(private clientOptions: ApiClientOptions) {}

  async init(): Promise {
    const headers: Record = {};

    // Strategy switch — algorithm selected once at init, transparent to callers
    switch (this.clientOptions.authType) {
      case AuthType.BASIC:
        const creds = Buffer.from(`${this.clientOptions.username}:${this.clientOptions.password}`).toString('base64');
        headers['Authorization'] = `Basic ${creds}`;
        break;
      case AuthType.BEARER:
        headers['Authorization'] = `Bearer ${this.clientOptions.token}`;
        break;
      case AuthType.API_KEY:
        headers[this.clientOptions.apiKeyHeaderName!] = this.clientOptions.apiKey!;
        break;
      case AuthType.CUSTOM:
        // headers injected via customHeaders below
        break;
    }

    // Additive merge — lets auth coexist with extra custom headers (e.g. Store: nz)
    if (this.clientOptions.customHeaders) {
      Object.assign(headers, this.clientOptions.customHeaders);
    }

    this.context = await request.newContext({ baseURL: this.clientOptions.baseURL, extraHTTPHeaders: headers });
  }
}

Decorator Pattern
Extending capability without modifying the base
What it is

ApiClientExt extends ApiClient and wraps each HTTP method with a *WithWrapper variant that returns an ApiResponseWrapper instead of a raw APIResponse. The base client is unchanged and still available for simple use-cases.

Why it matters

The raw ApiClient is fine for simple status checks. The decorated layer adds assertion chaining without touching the base. You can use either layer depending on how much depth a test needs.

Interview talking point
ApiClientExt extends ApiClient and decorates its HTTP methods. getWithWrapper calls the underlying GET but wraps the result in ApiResponseWrapper, which adds assertion chaining like assertJsonPath('user.id', 1). The base class stays minimal; the extended layer adds capability without modifying it.
Framework code
src/api/ApiClientExt.ts — decorator wrapping each HTTP verb
ApiClientExt — wraps each method to return ApiResponseWrapper
import { ApiClient } from './ApiClient';
import { ApiResponseWrapper } from './ApiResponse';

export class ApiClientExt extends ApiClient {
  // Calls the base GET, wraps the result — adds capability, doesn't modify ApiClient
  async getWithWrapper(url: string, queryParams?: Record): Promise {
    const response = await this.get(url, queryParams);
    return new ApiResponseWrapper(response);
  }

  async postWithWrapper(url: string, data?: unknown, headers?: Record): Promise {
    const response = await this.post(url, data, headers);
    return new ApiResponseWrapper(response);
  }

  async putWithWrapper(url: string, data?: unknown, headers?: Record): Promise {
    const response = await this.put(url, data, headers);
    return new ApiResponseWrapper(response);
  }

  async deleteWithWrapper(url: string, data?: unknown): Promise {
    const response = await this.delete(url, data);
    return new ApiResponseWrapper(response);
  }
}
5
Fluent Interface / Method Chaining
Assertion DSL that reads like an English spec
What it is

ApiResponseWrapper exposes assertion methods (assertStatus, assertJsonPath, assertHasHeader, …) each independently awaitable. The result: each test step reads as a declarative statement of intent rather than raw extraction + expect() calls.

Why it matters

Assertion logic is centralised in one class. Test code is shorter and intent-revealing. When an assertion method needs updating (e.g. better error messages), you fix it once.

Interview talking point
ApiResponseWrapper is a fluent assertion DSL. Instead of parsing JSON, extracting a field, and calling expect() separately, the test says await response.assertJsonPath('user.id', 1). The intent is immediately readable and the implementation stays in one place. It's similar to how Playwright's own expect(locator) API works.
Framework code

test('TC_01 - Should fetch booking by ID', async ({ apiClientExt }) => {
  const logger = createTestLogger('TC_01 Fetch booking by ID');

  logger.step('Step 1 - GET /booking/1');
  const response = await apiClientExt.getWithWrapper('/booking/1');

  logger.step('Step 2 - Assert response');
  // Fluent chain — each method is internally a full expect() assertion
  await response.assertStatus(200);
  await response.assertJsonPath('firstname', 'Jim');
  await response.assertJsonPath('depositpaid', true);
  await response.assertHasHeader('content-type');

  logger.step('Step 3 - Extract and log for report');
  const name = await response.extract('firstname');
  logger.verify('Firstname matches', 'Jim', name);
});

Fixture Pattern (Dependency Injection)
Playwright's test.extend() as an IoC container for page objects
What it is

base-test.ts uses test.extend<CustomFixtures>() to register all page objects. A test declares what it needs in its parameter list — Playwright constructs the object, runs the test, then runs teardown. The test never calls new EcommercePDPPage(page).

Why it matters

Teardown logic (e.g. the Firefox about:blank workaround) lives in the fixture, not scattered across 15 afterEach hooks. Tests are simpler. Lifecycle management is centralised.

Interview talking point
Playwright's fixture system is inversion of control for tests. A test that needs ecommercePDPPage just declares it as a parameter — the framework constructs the page object, runs the test, then executes teardown. The Firefox service-worker teardown workaround lives in the fixture body so it fires reliably on every teardown, not in individual afterEach blocks.
Framework code
src/config/base-test.ts — fixture registration with colocated teardown
base-test.ts — fixture lifecycle: construct → yield → teardown
export const test = base.extend({
  // Simple fixture: construct and yield
  homePage: async ({ page }, use) => {
    await use(new HomePage(page));
  },

  // Fixture with teardown colocated: Firefox workaround fires after every test
  ecommercePDPPage: async ({ page }, use) => {
    await use(new EcommercePDPPage(page));

    // Firefox teardown — prevents Juggler protocol hangs on SPA service workers.
    // This fires after EVERY test that uses ecommercePDPPage, automatically.
    // Without the fixture system, this would need an afterEach in every spec file.
    if (page.context().browser()?.browserType().name() === 'firefox') {
      await page.goto('about:blank', { waitUntil: 'commit', timeout: 5000 }).catch(() => {});
    }
  },

  // Fixture with injected dependency: softAssert needs a TestLogger seeded from test title
  softAssert: async ({}, use) => {
    await use(new SoftAssertHelper(new TestLogger(base.info().title)));
  },
});

Factory Method Pattern
Function-valued fixtures that create custom client instances
What it is

The API fixture system exposes createClient, createClientExt, createGraphQLClient — these are functions a test calls to get a differently-configured client instance (different auth, different base URL). Construction logic is centralised; the test only specifies what's different.

Why it matters

Most tests use the default apiClientExt fixture. Tests that need a second client with different auth (e.g. testing a different brand store) call the factory without reimplementing construction boilerplate.

Interview talking point
The createGraphQLClient fixture is a Factory Method: it's a function-valued fixture that tests call with overrides to get a custom client instance. The construction logic — init, context creation, error handling — is centralised in the factory. The test just says what's different, not how to build it.
Framework code

export interface ApiTestFixtures {
  // Regular fixtures — pre-built instances
  apiClient: ApiClient;
  apiClientExt: ApiClientExt;

  // Factory fixtures — functions that create customised instances on demand
  createClient: (options: Partial) => Promise;
  createClientExt: (options: Partial) => Promise;
  createGraphQLClient: (options?: Partial) => Promise;
}

export const apiTest = base.extend({
  // Factory fixture implementation
  createGraphQLClient: async ({ graphqlURL }, use) => {
    const clients: GraphQLClient[] = [];
    const factory = async (options?: Partial): Promise => {
      const client = new GraphQLClient({ baseURL: graphqlURL, ...options });
      await client.init();
      clients.push(client);  // tracked for teardown
      return client;
    };
    await use(factory);
    // Teardown: dispose all clients the test created
    for (const c of clients) await c.dispose();
  },
});

Registry Pattern (Static Token Store)
Class-level named store for cross-test token sharing
What it is

ApiClient.tokenStore is a static property — a class-level registry of named tokens. After a login step stores a token via ApiClient.storeToken('admin', token), any later test in the same worker retrieves it with ApiClient.getToken('admin').

Why it matters

API test suites authenticate once in beforeAll and reuse the token across many tests. Without this, every test would reauthenticate, adding round-trips and creating timing issues. Scoped to the worker process so there's no cross-spec leakage.

Interview talking point
The static tokenStore on ApiClient is a Registry pattern. It's class-level state shared across all instances in a worker process. One test logs in and registers the token; subsequent tests retrieve it by name. It's deliberately worker-scoped — Playwright isolates each spec file to its own worker, so there's no cross-suite leakage.
Framework code
src/api/ApiClient.ts — static tokenStore + helper methods
ApiClient — Registry: class-level storage, named lookup, factory helper
export class ApiClient {
  // static = one shared registry across all ApiClient instances in this worker
  private static tokenStore: Record = {};

  // Register a token under a logical name (e.g. 'admin', 'customer', 'pla-au')
  static storeToken(key: string, token: string): void {
    ApiClient.tokenStore[key] = token;
  }

  // Retrieve by name — returns undefined if not yet stored
  static getToken(key: string): string | undefined {
    return ApiClient.tokenStore[key];
  }

  // Factory helper: build a pre-authenticated client from a stored token
  static async withStoredToken(options: ApiClientOptions, tokenKey: string): Promise {
    const token = ApiClient.getToken(tokenKey);
    if (!token) throw new Error(`No token found for key: ${tokenKey}`);
    const client = new ApiClient({ ...options, authType: AuthType.BEARER, token });
    await client.init();
    return client;
  }
}

const token = (await loginRes.getData()).generateCustomerToken.token;

// Later test: retrieve and build authenticated client

  { baseURL: site.baseURL },
  'customer'

Typed Data Object Pattern
Explicit interfaces on every data shape — no inference reliance
What it is

Every data module in src/data/ declares TypeScript interfaces for every shape, then annotates constants and generator return types with those interfaces explicitly. TypeScript's inferred types are never relied on for exported test data.

Why it matters

Inferred types are invisible contracts. If someone renames a field in a data object, TypeScript catches every broken reference at compile time — npm run lint — rather than at runtime when the test silently passes with undefined.

Interview talking point
Test data is separated from test logic and fully typed with explicit interfaces. We declare AdminDataShape, then annotate the constant: export const AdminData: AdminDataShape = {...}. TypeScript enforces the contract — rename a field and every broken reference is a compile error, not a silent undefined at runtime.

Decorator-as-Logger (Cross-Cutting Concern)
Assertion + structured log entry in a single call
What it is

SoftAssertHelper wraps Playwright's expect.soft() and calls logger.verify() internally, both in a single method call. The caller gets the assertion and a structured log entry labelled 🔵 [SOFT] without writing both separately.

Why it matters

The old pattern required a manual logger.verify() call before expect.soft() — which caused double-logging and was easy to forget. SoftAssertHelper cross-cuts the logging concern so it's automatic and consistent. Similar to Aspect-Oriented Programming's advice weaving.

Interview talking point
SoftAssertHelper is a thin Decorator over Playwright's expect.soft. It cross-cuts two concerns — assertion and logging — into a single call. Every soft assertion is automatically logged with a consistent format. The test author can't accidentally forget to log it, and can't double-log it either. It's similar to the idea of AOP advice: behaviour woven in without changing the caller.
Framework code
src/utils/soft-assert-helper.ts — cross-cutting assert + log
SoftAssertHelper — each method asserts AND logs in one call
export class SoftAssertHelper {
  constructor(private readonly logger: TestLogger) {}

  // Both assert and log happen here — caller does neither separately
  toBe(actual: unknown, expected: unknown, message?: string): void {
    // 1. Log with [SOFT] tag to the structured test report
    this.logger.verify(message ?? `Value should be ${String(expected)}`, expected, actual, true);
    // 2. Soft assert — test continues past failure
    expect.soft(actual, message).toBe(expected);
  }

  toBeGreaterThan(actual: number, expected: number, message?: string): void {
    this.logger.verify(message ?? `Value should be > ${expected}`, `> ${expected}`, actual, true);
    expect.soft(actual, message).toBeGreaterThan(expected);
  }

  // Async variant for Playwright Locator assertions
  async toBeVisible(locator: Locator, message?: string): Promise {
    this.logger.verify(message ?? 'Element should be visible', 'visible', '?', true);
    await expect.soft(locator, message).toBeVisible();
  }
}

// ── Usage in a test ───────────────────────────────────────────────
test('E2E-PDP-001', async ({ ecommercePDPPage, softAssert }) => {
  const productName = await ecommercePDPPage.getProductName();

  // ONE call → assertion + log entry. Never call logger.verify() before softAssert.*
  softAssert.toBeGreaterThan(productName.length, 0, `${site.name}: Product name non-empty`);
  softAssert.toBe(await ecommercePDPPage.isImageGalleryVisible(), true, 'Gallery visible');
  // Test continues past any failure — both assertions collected and reported together
