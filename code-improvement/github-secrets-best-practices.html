GitHub Secrets Best Practices

Playwright TypeScript Framework — Environment Variable Security Guide — June 2026

The Problem

⚠ Critical Security Finding The .env.testing, .env.staging, .env.production, and .env.docker files were tracked in git — confirmed by git ls-files. No .gitignore rule existed to exclude them. Since the repository is public, any credentials in those files were publicly accessible in the commit history.

The framework's src/config/environment.ts loads config via dotenv.config(). Because dotenv does not override already-set process.env values, injecting secrets via workflow env: blocks takes precedence over .env.* files automatically — so no code changes to environment.ts are required.

The Hybrid Model

GitHub provides three mechanisms for supplying configuration to CI. The correct solution uses all three:

GitHub Environment Secrets
→
Per-env sensitive values
(same key, different value per env)
Repository Secrets
→
Infra tokens shared across all envs
(Cloudflare, Slack, Percy)
Environment Variables (vars)
→
Non-sensitive per-env config
(URLs, DB names — visible in logs)
Hardcoded defaults
→
Timeout, workers, headless
(already CI-aware in environment.ts)

Environment Secrets

  • Same key name, different value per environment
  • Only accessible when job declares environment: <name>
  • Values are masked in logs
  • Right for: passwords, tokens, API keys that differ per env

Repository Secrets

  • Single value, accessible to all jobs in the repo
  • Does not require environment: on the job
  • Values are masked in logs
  • Right for: infra tokens that don't change per env

Environment Variables (vars)

  • Like secrets but visible in logs
  • Requires environment: on the job (same as secrets)
  • Right for: URLs, hostnames, DB names — non-sensitive config

Fork PR Restriction

  • Public repos: fork PRs receive no secrets
  • Jobs with secret dependencies should gate on github.event.pull_request.head.repo.full_name == github.repository
  • Prevents silently-empty-credential runs from fork contributors
How dotenv + GitHub Secrets interact dotenv.config() calls Object.assign only for keys NOT already present in process.env. When a GitHub Actions env: block sets a variable, it's already in process.env before the test runner starts — so the workflow value wins over the .env.* file value. No code changes needed.

Step 0 — Rotate all credentials first Before configuring GitHub secrets, rotate every password, API key, and token that was stored in the .env.* files. The git history contained them in a public repo — treat all values as compromised.

Go to Settings → Environments → New environment and create three environments:

Untrack & Purge Git History

Do this only after wiring secrets in GitHub and validating a workflow run succeeds. Untracking breaks the local .env.* fallback. If secrets aren't wired yet, tests will get empty credentials.
  1. Validate secrets are wired

    Run playwright.yml via workflow_dispatch against testing. Confirm the logs show injected URLs (not the fallback defaults from environment.ts like demo.nopcommerce.com).

  2. Untrack the env files
    git rm --cached .env.testing .env.staging .env.production .env.docker
  3. Commit the untrack + gitignore
    git add .gitignore .env.example
git commit -m "security: untrack env files and add gitignore rule"
  4. Install git-filter-repo (if not already installed)
    python -m pip install git-filter-repo
  5. Purge the files from all history
    git filter-repo \
  --path .env.testing \
  --path .env.staging \
  --path .env.production \
  --path .env.docker \
  --invert-paths \
  --force

    This rewrites every commit. All commit SHAs change. Any existing clone will be out of sync.

  6. Re-add remote and force-push

    git filter-repo removes remotes as a safety measure. Re-add and push:

    git remote add origin https://github.com/hoanglk12/playwright-typescript.git
git push --force origin main
  7. Verify history is clean
    git log --all -p -- .env.testing .env.staging .env.production .env.docker

    This should return no output. If it returns commits, the purge did not run successfully.

Syncing Other Machines

After a history rewrite + force-push, every existing clone has diverged history. Any git pull will fail or produce incorrect results.

Option A — Fetch + Reset (keeps local uncommitted work)

git fetch origin
git reset --hard origin/main

Option B — Re-clone (cleanest)

cd ..
rm -rf playwright-typescript
git clone https://github.com/hoanglk12/playwright-typescript.git
cd playwright-typescript
Local .env files After syncing, recreate your local .env.testing from .env.example — the tracked file is gone from the repo. Copy .env.example and fill in the values from your password manager or 1Password.

FAQ

Why not just use repository secrets for everything?

Repository secrets store a single value per key. You cannot have PASSWORD=testpass for testing and PASSWORD=stagingpass for staging using the same repository secret. GitHub Environment secrets solve this: same key name, different value per named environment — resolved based on which environment the job declares.

Do I need to change environment.ts?

No. dotenv.config() only fills in keys that are not already in process.env. GitHub Actions injects env: block values before the runner starts, so those win automatically.

What happens if a secret is not set?

The expression ${{ secrets.MY_SECRET }} resolves to an empty string when the secret doesn't exist. The test runner will then fall back to the hardcoded default in environment.ts (e.g. testpassword, demo.nopcommerce.com). Tests will likely fail — which is a useful signal that the secret is missing, rather than silently testing against wrong infrastructure.

Can I commit .env.example?

Yes — that's the point. It contains all key names but no values. The .gitignore rule is .env* with a !.env.example exception, so the example file is explicitly allowed.

What does git filter-repo do to commit history?

It rewrites every commit that touched the specified paths, producing new commit SHAs. The old commits (with the .env.* files) are no longer reachable from any branch. GitHub will garbage-collect them after the force-push. Note: GitHub's cache may retain them briefly — if you need an immediate guarantee, contact GitHub Support to clear the cache.

Does the production environment require extra setup?

Yes. In Settings → Environments → production, enable Required reviewers and add at least one approver. Any workflow job that declares environment: production will pause and wait for manual approval before running. This prevents accidental production runs from push triggers.
